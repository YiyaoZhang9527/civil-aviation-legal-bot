# 民航法律机器人一致性改进设计方案

> 基于一致性测试（10轮）和学术调研（22篇论文/项目）的深度分析
> 日期: 2026-05-31
> 状态: 待选择

---

## 背景摘要

10轮一致性测试核心发现：
- 证据组合10轮10种完全不同，没有一条法条在所有轮次中都出现
- 结论严格一致率60%，1轮严重偏离（检索到完全不相关的空管规则）
- 引用校验同一条法条在不同轮次被判为 supported/partial/unsupported 都有
- Reflexion自检每轮触发满额2轮，从未通过
- 答案格式只有3/10轮有可解析的法条引用

学术调研核心结论：
- LLM在法律问题上存在10%-50%的不稳定性（ACM 2026实证）
- temperature=0不等于确定性，API服务端仍有非确定性
- RAG商用法律工具仍有17%-33%的幻觉率（Stanford 2024）
- 共识：检索路由不应依赖LLM，引用校验不应依赖LLM

---

## 方案总览

共5个方向、12个可选方案。按改进层级分层：

```
Layer 1: 检索层（消除50%的不确定性）    → 方案A1, A2, A3
Layer 2: 融合截断层（消除20%的不确定性） → 方案B1, B2, B3
Layer 3: 校验层（消除20%的不确定性）    → 方案C1, C2, C3
Layer 4: 缓存层（消除10%的不确定性）    → 方案D1, D2
Layer 5: 格式与自检层                   → 方案E1
```

## A. 检索层改进方案（消除核心不确定性源头）

### 方案 A1: 查询确定性门控 (Query Deterministic Gate)

**核心理念:** 原始用户问题永远参与检索，LLM改写仅作为可选扩展。堵住不确定性源头。

**论文依据:**
- Nexumo "When Smart Query Rewriting Breaks Your RAG"：query expansion是条件有益的，不是普遍有益的。Expansion Gate架构对每个query判断是否展开。
- MUE/DUE论文的Evidence Gate：相似度高的句子不一定能作为证据。

**代码诊断:**
`agents.py` RewriteAgent(行344-380)每次LLM调用产生不同queries/law_hints/article_hints，这些是检索管线的唯一入口。`RetrievalAgent.retrieve()`(行447-513)对每个query独立调用`search_index_tree()`，多query结果拼接后取top-8——不同轮次各query召回集合不同，top-8自然不同。

**具体改动:**

1. 改 `RetrievalAgent.retrieve()` (agents.py:447-513)：
   - 第一个query固定为 `state.normalized_question`（原始问题归一化），不依赖LLM
   - LLM生成的queries仅作扩展，需通过门控

2. 新增 `_should_expand_query()` 函数：
```python
def _should_expand_query(original: str, expanded: str) -> bool:
    """判断扩展query是否保留原始约束。"""
    anchors = re.findall(r'《[^》]+》|CCAR[\-\d.]+|第[一二三四五六七八九十百千\d]+条', original)
    if not anchors:
        return True  # 无精确锚点，允许扩展
    for anchor in anchors:
        if anchor not in expanded:
            return False  # 丢失原始锚点，拒绝
    return True
```

3. 改 RewriteAgent prompt：要求改写时必须保留法名、条号、CCAR编号等精确词汇

**预期效果:**
- 原始query确定性，BM25+向量对它的排名完全确定
- 证据组合一致性从0%提升到60-70%（核心法条由primary query保底）
- 扩展query提供覆盖补充但不再主导结果集

**工程量:** 1.5人天

**风险:** 原始query召回可能不如LLM改写精确。缓解：原始query和扩展query共同参与RRF，原始query权重≥50%

### 方案 A2: Hint规则化检索 (Rule-Based Deterministic Retrieval)

**核心理念:** 把LLM产生的law_hints/article_hints从"加分项"升级为"确定性过滤器"，结合法条结构化元数据实现硬规则匹配。

**论文依据:**
- WhyHow.AI规则化检索：开发者定义规则（文件名+页码范围）自动构建metadata filter缩小搜索范围。"deterministic retrieval gives the peace of mind that a deterministic system in production brings."

**代码诊断:**
`retrieval.py` 的 `_score_hints()`(行220-236) 当前hint只是加分。`tree_retrieval.py` 的条级检索中 hint_scores 传空字典 `{}`——**hint信号完全丢失**，这是现有bug。

**具体改动:**

1. 新增数据结构：
```python
@dataclass
class RuleFilter:
    law_ids: set[str]
    article_nums: set[str]
    def matches(self, law_id: str, node_id: str) -> bool:
        if self.law_ids and law_id not in self.law_ids:
            return False
        if self.article_nums:
            art_num = re.search(r"(\d+)", node_id)
            if art_num and art_num.group(1) in self.article_nums:
                return True
            return False
        return True
```

2. 改 `search_index_tree()` 检索流程：
   - 先构建RuleFilter → 精确直读已有法条
   - 然后走普通检索但用RuleFilter过滤候选集：只在规则指定的法内检索
   - 两部分合并，确定性证据优先
   - RuleFilter构建失败时降级为现有流程

3. 修复 `tree_retrieval.py` 中 hint_scores 传空字典的bug

**预期效果:**
- 当law_hints可解析时，检索范围从全量缩小到特定法，确定性大幅提升
- 证据组合一致性：hints可解析时从0%提升到90%+
- 修复hint信号在树检索中丢失的bug

**工程量:** 2人天

**风险:** LLM生成的law_hints本身有不确定性。缓解：对law_hints做模糊匹配容错 + 规则化检索与普通检索合并

### 方案 A3: MUE/DUE确定性证据门控

**核心理念:** 检索后、喂给LLM之前，加一层确定性证据准入检查。每个候选法条独立评估是否具备"证据效用"。"相似度不等于可用性"。

**论文依据:**
- MUE/DUE（arXiv:2603.18011）：三个确定性信号评估证据效用——语义相似度Sim、词汇覆盖Rel、语料集概念独特性CI。固定权重组合(λ=0.50, μ=0.30, ν=0.20)，通过Evidence Gate决定是否允许。论文关键洞察：法律领域很多条文提到同一概念，但只有少数条文明确陈述了回答问题所需的条件/规则/定义。

**具体改动:**

1. 新增 `legalbot/evidence_gate.py`：

```python
class EvidenceGate:
    def score_mue(self, unit: EvidenceUnit, query: str) -> float:
        """三信号加权：CI*0.50 + Sim*0.30 + Rel*0.20"""
        ci = self._conceptual_importance(unit.text)  # 语料集独特性
        sim = unit.sim_score                         # 向量相似度
        rel = self._lexical_relevance(unit.text, query)  # 词汇覆盖
        return 0.50 * ci + 0.30 * sim + 0.20 * rel

    def select_due(self, units, query, top_k=8):
        """迭代选择，抑制冗余。"""
        admitted = []
        for unit, score in sorted_units:
            if self._is_redundant(unit, admitted, threshold=0.85):
                continue
            if score >= self._mue_threshold(query):
                admitted.append(unit)
        return admitted

    def check_gate(self, evidence, query) -> bool:
        """Evidence Gate：至少一个anchor unit通过。"""
        has_anchor = any(
            self._lexical_relevance(u.text, query) >= 0.30
            and u.sim_score >= 0.35
            for u in evidence
        )
        return has_anchor
```

2. 在 `RetrievalAgent.retrieve()` 的Cross-Encoder精排前插入MUE/DUE选择

**预期效果:**
- DUE冗余抑制消除"同一概念多个条文互相挤占"问题
- Evidence Gate提供确定性锚点：通过门控的证据组合跨轮高度一致
- 证据组合一致性从0%提升到80%+，零LLM调用

**工程量:** 3人天

**风险:** 阈值需要标注数据调优。初期用论文推荐值(τ_rel=0.30, τ_sim=0.35)

## B. 融合截断层改进方案（消除边界不确定性）

### 方案 B1: 归一化凸组合替代RRF

**核心理念:** RRF丢弃分数分布信息，改用min-max归一化后的凸组合，保留分数距离信息。

**论文依据:**
- TM2C2论文（arXiv:2210.11934）：凸组合对归一化方式不敏感，仅需少量标注查询即可调优alpha参数。RRF基于排名丢弃分数分布信息，跨域泛化差。

**当前代码问题（agent深度阅读发现）:**
`hybrid_retrieval.py` 的 `rrf_fuse` 函数中，hint分支不走RRF排名而是直接注入分数 `w_hint * min(hint_s/12, 1.0)`——这是一个RRF与分数注入的混合体，设计不一致。此外 `tree_retrieval.py` 的条级检索中 hint_scores 传空字典 `{}`，hint信号完全丢失。

**具体设计:**

改 `hybrid_retrieval.py` 的 `rrf_fuse` 为 `convex_fuse`：

```python
def convex_fuse(
    bm25_results, vector_results, hint_scores
) -> dict[tuple[str, str], float]:
    """归一化凸组合融合：保留分数分布信息。"""
    # min-max归一化到[0,1]
    def norm(m):
        if not m: return {}
        vals = list(m.values())
        mn, mx = min(vals), max(vals)
        rng = mx - mn or 1.0
        return {k: (v - mn) / rng for k, v in m.items()}

    bm25_n = norm(dict(bm25_results))
    vec_n = norm(dict(vector_results))
    hint_n = norm(hint_scores) if hint_scores else {}

    w_bm25 = 1.0 - config.HYBRID_WEIGHT_VECTOR - config.HYBRID_WEIGHT_HINT
    scores = {}
    for key in set(bm25_n) | set(vec_n) | set(hint_n):
        s = (w_bm25 * bm25_n.get(key, 0.0)
             + config.HYBRID_WEIGHT_VECTOR * vec_n.get(key, 0.0)
             + config.HYBRID_WEIGHT_HINT * hint_n.get(key, 0.0))
        if s > 0:
            scores[key] = s
    return scores
```

同步修复 `tree_retrieval.py` 中 hint_scores 传空字典的bug。

**预期效果:**
- 保留分数距离信息，边缘法条不再因RRF排名截断被随机纳入/排除
- 修复hint信号在树检索中丢失的bug
- TM2C2论文实验显示NDCG提升2-5%

**工程量:** 2人天

**风险:**
- min-max归一化在单一候选时rng=0，需兜底
- 需要重新标定权重参数

### 方案 B2: 聚类自适应Top-K截断（CAR）

**核心理念:** 根据融合分数分布的自然断裂点动态决定top-k，简单查询取2-3条，复杂查询取10-15条。

**论文依据:**
- CAR论文（Coinbase生产系统，arXiv:2511.14769）：用BisectingKMeans聚类检测分数断崖点。Coinbase验证：token消耗减少60%，延迟降低22%，幻觉减少10%。每查询<50ms。

**具体设计:**

1. 新增文件 `legalbot/adaptive_cutoff.py`（纯函数，约80行）
2. 在融合函数调用后插入自适应截断
3. 替代 `config.py` 中的固定 `DEFAULT_TOP_K = 8`

**预期效果:**
- 简单查询：减少噪音法条注入，降低幻觉率
- 复杂查询：扩大召回范围，边缘法条不再被硬截断
- 彻底消除固定top-k的不确定性

**工程量:** 3人天（sklearn已有间接依赖，BSD许可可商用）

**风险:**
- 聚类在短列表（<6个候选）时不稳定，需设min_k下限
- 需要参数调优（聚类数搜索范围、位置惩罚权重）

### 方案 B3: 信号置信度加权截断

**核心理念:** 不引入聚类算法，而是看BM25、向量、hint三个信号中有几个认为该候选相关。取多信号一致的候选，确定性截断。

**论文依据:**
- ReproRAG：不同信号对同一查询的top-50重叠率仅43-54%，持久结果的排名完全稳定
- Stable-RAG：排列不变性是核心问题

**具体设计:**

```python
def confidence_cutoff(bm25_results, vector_results, hint_scores,
                      min_signals=2, max_k=15):

    bm25_keys = {k for k, _ in bm25_results[:30]}
    vec_keys = {k for k, _ in vector_results[:30]}
    hint_keys = {k for k, s in hint_scores.items() if s > 0}

    result = []
    for key in bm25_keys | vec_keys | hint_keys:
        signals = sum([key in bm25_keys, key in vec_keys, key in hint_keys])
        if signals >= min_signals:
            result.append((key, signals))  # 确定性：同信号永远同结果
    result.sort(key=lambda x: x[1], reverse=True)
    if len(result) < 3:

        ...
    return result[:max_k]
```

**预期效果:**
- 确定性截断：同一查询同一信号，截断结果完全确定
- 多信号一致的候选比单信号可靠得多
- 零外部依赖，实现极简

**工程量:** 1.5人天

**风险:**
- min_signals=2对极端边缘法条可能过于严格，需设fallback

## C. 校验层改进方案（消除LLM判断不确定性）

### 方案 C1: 确定性Cross-Encoder校验替代LLM校验

**核心理念:** 把"这条证据是否支持这个claim"从LLM判断改为cross-encoder语义相似度计算+阈值判定。LLM只负责claim抽取，不做校验判断。

**论文依据:**
- regulated-rag："Citation grounding belongs in code, not in prompts." 引用锚定和低置信度拒绝在代码层面强制执行。
- 4xxi三层引用验证：第二层passage alignment用cross-encoder而非LLM计算语义对齐分数。
- Non-Determinism论文：API服务端LLM不可复现，但本地cross-encoder推理是确定性的。

**当前代码问题:**
`CitationVerifier.verify()` 把claim抽取+证据校验+置信度打分全塞进一次LLM调用。同一条法条在不同轮次被判为不同状态，因为LLM在temperature=0下对同一prompt仍有token-level概率波动。ReflexionAgent又用另一个LLM评估CitationAgent的结果——用随机性校验随机性，永远不会收敛。

**具体设计:**

1. 改 `citation.py` 的 `CitationVerifier` 类，重写 `verify()` 为三阶段：

```
Stage 1: LLM claim抽取（保留现有extract_claims，这是LLM唯一参与点）
Stage 2: 确定性语料库存在性检查（零LLM，检查node_id是否在索引中）
Stage 3: 对每个claim用已有的bge-reranker-v2-m3计算cross-encoder得分（确定性数值计算）
Stage 4: 基于阈值判定status（supported>=0.65, partial>=0.40, unsupported<0.40）
```

2. ReflexionAgent的影响：
   - 快速路径真正可用了——cross-encoder得分是确定性的，`all_supported and confidence>=0.7`不再有随机波动
   - ReflexionAgent的LLM调用可以保留但仅作为补充审计

**预期效果:**
- 同一法条跨轮次判定结果一致性：从约30%提升到**100%**
- ReflexionAgent快速路径触发率：从约0%提升到60-70%
- 单次校验延迟：降低约40%（省去LLM调用，cross-encoder本地推理约50ms/条）

**工程量:** 3人天

**风险:**
- 阈值（0.65/0.40）需要用真实标注数据调优
- bge-reranker-v2-m3是通用模型，对法律条文判别力可能不如领域微调模型。但目标是确定性而非完美精度。

### 方案 C2: 多维度投票仲裁（ConsJudge启发）

**核心理念:** 不依赖单次LLM判断，从多个评估维度独立生成判断，用一致性分数做仲裁。分裂时降级到cross-encoder。

**论文依据:**
- ConsJudge（ACL 2025）：用k个不同评估维度组合生成k个判断，计算判断间一致性，一致性最高的被采纳。

**具体设计:**

1. 新增 `ConsensusCitationVerifier` 类
2. 三个维度独立判断：faithfulness（忠实度）、relevance（相关性）、sufficiency（充分性）
3. 投票仲裁：2/3多数决定结果；三方分裂时降级到cross-encoder

**预期效果:**
- 单维度翻转不再影响最终结果，跨轮次一致性提升到80%+
- 分裂率约10-15%，降级到cross-encoder后最终一致性100%

**工程量:** 4人天

**风险:**
- LLM调用量增大3倍（三维度），但可并行
- 两个方案可组合：C1处理evidence侧确定性，C2处理claim侧质量

### 方案 C3: Salient Point索引 + 命题级结构化匹配

**核心理念:** 索引构建阶段为每条法条预提取核心法律命题（salient points），校验时做claim与命题的结构化匹配。

**论文依据:**
- Ark Legal的Proposition Cross-Check：索引阶段预提取每个法条的salient points，校验时做命题级匹配。
- regulated-rag v0.2计划：从set-membership升级到predicate-presence check。

**具体设计:**

1. `types.py` 中 `LawNode` 增加 `salient_points: list[str]` 字段
2. 索引构建时LLM一次性提取每个法条的核心法律命题并缓存
3. 运行时用cross-encoder做claim-point精确匹配，零LLM调用

**预期效果:**
- 校验粒度从"整条法条"提升到"具体命题"
- 运行时零LLM调用，延迟极低
- 为法条冲突检测提供结构化基础

**工程量:** 5人天

**风险:**
- salient points提取质量需人工抽检验证
- 索引构建时间增加（每条法条一次LLM调用，一次性成本）

## D. 缓存层改进方案

### 方案 D1: 证据门控答案缓存 (Evidence-Gated Answer Cache)

**核心理念:** 将GroundedCache的四门控路由思想移植到多Agent管线中，为LegalOrchestrator增加答案缓存层。用已有的CitationAgent结果替代论文的G4词法检查。

**论文依据:**
- GroundedCache（arXiv:2605.27494）：四门控串行链——G1查询相似度、G2证据重叠度(Jaccard on chunk hashes)、G3源版本一致性、G4词法支撑度。核心发现：**G4词法支撑检查承担了绝大部分安全增益**（消融实验USR +0.125），其余门控以近零成本提供纵深防御。检索侧缓存比输出侧缓存安全得多。

**代码诊断:**
当前系统**零缓存**，每次请求都完整执行全链路（11-12次LLM调用）。CitationAgent的校验结果只传给SynthesisAgent组织答案，没有反馈到任何缓存或检索决策。法规库更新后无感知机制，旧答案可能引用过期法条。

**具体设计:**

1. `types.py` 新增：
```python
@dataclass
class CacheEntry:
    query_normalized: str
    query_embedding: list[float]
    evidence_signature: str    # SHA-1(law_id:node_id:chunk_hash) sorted & joined
    answer: str
    citations: list[CitationCheck]
    evidence: list[Evidence]
    timestamp: float
```

2. `LegalOrchestrator` 新增缓存路由：
```python
def _answer(self, question):
    # 缓存路由
    cached = self._cache_lookup(question)
    if cached:
        # G4: 用CitationAgent校验结果作为支撑度门控
        supported_ratio = sum(
            1 for c in cached.citations if c.status in {"supported", "partial"}
        ) / max(len(cached.citations), 1)
        if supported_ratio >= 0.6:
            return AnswerResult(answer=cached.answer, status="cache_hit", ...)
    # 原有全链路
    result = self._answer_full(question)
    self._cache_insert(question, result)
    return result

def _compute_evidence_signature(self, evidence):
    """内容寻址签名——法规更新后自动失效"""
    import hashlib
    parts = [f"{e.law_id}:{e.node_id}:{hashlib.sha1(e.text.encode()).hexdigest()[:12]}"
             for e in sorted(evidence, key=lambda x: (x.law_id, x.node_id))]
    return hashlib.sha1("|".join(parts).encode()).hexdigest()
```

**预期效果:**
- 对重复/近似问题（航班延误、行李丢失等高频咨询），p95延迟降低70-80%
- 首次查询零开销（仅多一次embedding计算+字典查找）
- 法规库更新后证据签名自动失效，不返回过期答案

**工程量:** 2人天（约150-200行新增代码，不改变现有Agent接口）

**风险:** 缓存阈值(0.92相似度、0.6支撑率)需实测调优。需加LRU淘汰。

### 方案 D2: 检索侧查询缓存 (Query-Conditioned Retrieval Cache)

**核心理念:** 比D1更轻量——不缓存答案，只缓存检索结果。相同/相似问题复用evidence列表，跳过检索+校验+自检，直接进SynthesisAgent。

**论文依据:**
- GroundedCache明确指出：**检索侧缓存（Proximity/QVCache）比输出侧缓存（GPTCache）安全得多**——缓存命中时仍会重新生成答案，只是跳过检索环节，因此不可能返回过期答案。
- Proximity论文：缓存检索结果而非输出，验证检索侧缓存更安全。

**具体改动:**

1. 新增 `legalbot/query_cache.py`：
```python
class RetrievalCache:
    def lookup(self, question: str, threshold: float = 0.92) -> list[Evidence] | None:
        q_vec = self.vector_model.encode(question)
        for entry in self._entries:
            if cosine_similarity(q_vec, entry["vec"]) >= threshold:
                return entry["evidence"]
        return None
```

2. 在 `_answer()` 中检索前查缓存。命中则跳过：RewriteAgent → RetrievalAgent → CitationAgent → ReflexionAgent，直接进SynthesisAgent

**预期效果:**
- 比D1更安全——缓存命中时仍重新生成答案，不存在过期风险
- 延迟降低约50%（跳过检索+校验+自检，只做一次LLM生成）
- 工程量更小

**工程量:** 1.5人天

**风险:** 需要确认embedding接口可复用检索模块已有的sentence-transformers

## E. 格式与自检层改进方案

### 方案 E1: 确定性词法支撑自检 + 答案格式后处理

**核心理念:** 两部分：(1) Reflexion循环中加入GroundedCache G4门控的中文适配版——答案中的关键法律术语必须出现在证据文本中，不满足则直接判定gap，省掉LLM调用；(2) 答案后处理强制附加标准引用列表。

**论文依据:**
- GroundedCache G4门控：`S(a,C) = |τ(a) ∩ τ(C)| / |τ(a)|`，词法支撑检查承担了绝大部分安全增益。关键特性：**确定性、可审计、无LLM judge二次噪声**。
- GroundedCache论文消融实验：移除G4后USR上升0.125，是四个门控中安全贡献最大的。

**具体改动:**

**Part 1: ReflexionAgent确定性门控** (改agents.py ReflexionAgent类)

def normalize_citations(answer: str, evidence: list[Evidence]) -> str:
    """后处理：在答案末尾附加标准引用列表。"""
    citation_block = "\n\n【引用来源】\n"
