> 调研日期: 2026-05-31

> 调研目标: 搜索学术界和工业界对 RAG 输出一致性、法律AI可复现性、检索确定性方案的研究成果

## 三、论文与项目详情

### 3.1 RAG系统一致性研究

#### [1] ReproRAG: On The Reproducibility Limitations of RAG Systems

- **来源:** 华盛顿大学 & 太平洋西北国家实验室, 2025
- **URL:** https://arxiv.org/html/2509.18869v1
- **核心贡献:** 系统性地量化了RAG管线中不确定性的来源层次
- **关键发现:**
  - embedding模型选择和动态数据插入是最大的不一致性来源
  - ANN检索算法（HNSW、IVF）在受控环境下可以实现完美可复现性
  - 提出了 Exact Match Rate、Jaccard Similarity、Kendall's Tau 三种量化指标
- **已开源**
- **对我们的启示:** 我们的BM25+向量RRF融合中，向量部分本身是确定性的，不确定性来自LLM生成的queries不同导致BM25命中不同

#### [2] Con-RAG: Improving Consistency in Retrieval-Augmented Systems with Group Similarity Rewards

- **来源:** arXiv, 2025
- **URL:** https://arxiv.org/abs/2510.04392
- **核心贡献:** 将RAG一致性分解为检索层、生成层、端到端三个维度
- **关键发现:**
  - 提出PS-GRPO（Paraphrased Set GRPO）强化学习方法
  - 通过同一查询的多个改写版本生成多组输出，用组间相似度作为奖励信号训练生成器
  - 在Llama3.1和Qwen2.5上验证，显著提升一致性和准确率
- **对我们的启示:** 三维一致性分解框架可直接用于评估我们的系统——检索层0分、生成层低分、端到端60分

#### [3] RAGdeterm: Deterministic Retrieval-Augmented Generation for Code

- **来源:** ScienceDirect / SSRN, 2026
- **URL:** https://www.sciencedirect.com/science/article/pii/S2352711026001299
- **核心贡献:** 完全抛弃embedding和向量数据库，改用关系数据库建模结构依赖关系
- **关键发现:**
  - 确定性检索：相同输入始终产生相同检索结果
  - 显著减少prompt大小和计算成本
  - 可复现、可验证
- **对我们的启示:** 法律文本也有结构依赖（法→章→条），可以用类似的关系建模替代纯语义检索

#### [4] Controlled Multi-Hop RAG: A Deterministic Parallel Pipeline

- **来源:** Research Square, 2025
- **URL:** https://www.researchsquare.com/article/rs-9148928/latest.pdf
- **核心贡献:** 提出确定性并行管线架构作为Agent式检索的替代方案
- **关键发现:**
  - 用确定性管线取代不可预测的LLM自主规划
  - 在生产级RAG平台MIKA中实现
  - 强调可控性和可审计性
- **对我们的启示:** 我们的多Agent编排（SubjectAgent→IssueAgent→RewriteAgent→...）就是论文批评的"不可预测的LLM自主规划"模式

#### [5] CacheRAG: A Semantic Caching System for RAG in KGQA

- **来源:** arXiv, 2026
- **URL:** https://arxiv.org/html/2604.26176v1
- **核心贡献:** 将无状态的LLM规划器转化为持续学习者
- **关键发现:**
  - 通过中间语义表示（ISR）解耦自然语言与知识图谱schema
  - 两层分层索引+MMR实现多样性缓存检索
  - 确定性有界扩展算子避免无界执行
  - CRAG数据集上准确率提升13.2%，真实性提升17.5%
- **对我们的启示:** 缓存是解决重复查询的有效手段

#### [6] How Do I Build a Deterministic RAG Pipeline?

- **来源:** Charles Chen Wiki
- **URL:** https://wiki.charleschen.ai/ai/processed/wiki/llm-core/rag/queries/security/how-to-build-deterministic-rag-pipelines
- **核心贡献:** 定义了RAG管线不确定性的六个独立层
- **关键发现:**
  - 六层不确定性：embedding、检索算法(HNSW)、排名打分(浮点tie-breaking)、上下文组装、LLM采样、缓存行为
  - 三级确定性标准：实用级(99%+, 1小时配置)、强确定性(可复现, 1-2天)、比特级(合规审计, 1-2周)
  - **temperature=0 远远不够**：GPU浮点运算顺序、索引更新、排名平局都是独立的不确定性来源

#### [7] Beyond Vector Search: Building a Deterministic 3-Tiered Graph-RAG System

- **来源:** Machine Learning Mastery
- **URL:** https://machinelearningmastery.com/beyond-vector-search-building-a-deterministic-3-tiered-graph-rag-system/
- **GitHub:** https://github.com/mmmayo13/rag_facts_quadstore
- **核心贡献:** 三级优先级数据架构
- **关键发现:**
  - Priority 1: 知识图谱(QuadStore, SPOC四元组格式)存储原子事实——绝对真相层
  - Priority 2: 统计聚合数据
  - Priority 3: 向量数据库仅作为兜底
  - prompt内嵌优先级冲突解决规则，让LLM确定性地裁决事实冲突
- **对我们的启示:** 法律文本天然适合知识图谱建模——法律条文就是"原子事实"，不应该用模糊的向量检索作为主要检索手段

### 3.2 法律AI可复现性研究

#### [8] LLMs Provide Unstable Answers to Legal Questions

- **来源:** ACM, 2026
- **URL:** https://dl.acm.org/doi/full/10.1145/3769126.3769245 / https://arxiv.org/html/2502.05196v1
- **核心贡献:** 首个系统性的法律LLM不稳定实证研究
- **关键发现:**
  - 对500个真实法律案件问题，GPT-4o、Claude-3.5、Gemini-1.5在temperature=0下各调用20次
  - **10.6%~50.4%的问题出现不稳定**（同一问题不同运行给出完全相反的判决结论）
  - 不稳定主要来自模型本身的特性而非问题特征
  - 三个模型间的不稳定问题重合度很低
- **对我们的启示:** 我们的40%不稳定率(4/10轮偏离)在这个范围内，说明问题不在我们的实现，在LLM本身

#### [9] Non-Determinism of Deterministic LLM System Settings in Hosted Environments

- **来源:** Eval4NLP, 2025
- **URL:** https://aclanthology.org/anthology-files/pdf/eval4nlp/2025.eval4nlp-1.12.pdf
- **核心贡献:** 解释了为什么temperature=0仍然不稳定
- **关键发现:**
  - API调用的LLM在字符串级别几乎无法复现
  - 但在解析答案层面较为稳定
  - **本地部署的Llama3-8b无此问题**
  - 非确定性来自服务端工程优化（连续批处理、前缀缓存等）
- **对我们的启示:** CitationAgent的校验矛盾不是代码bug，是DeepSeek API服务端的非确定性。如果要求确定性，要么本地部署，要么把LLM判断换成规则判断

#### [10] LegalCiteBench: Evaluating Citation Reliability in Legal Language Models

- **来源:** arXiv, 2026
- **URL:** https://arxiv.org/html/2605.10186
- **核心贡献:** 法律LLM引用可靠性基准
- **关键发现:**
  - 24K评估实例，覆盖21个LLM
  - 最强模型（Claude Sonnet 4.5）引用检索得分不超过**7/100**
  - 20/21个模型在检索密集任务上的误导回答率（MAR）超过94%
  - **无外部检索锚定时，LLM不应被信任生成法律引用**
- **对我们的启示:** 我们的SynthesisAgent让LLM基于证据生成引用，方向正确。但引用格式遵守率只有30%，需要加强格式约束

#### [11] Stanford "AI on Trial" -- RAG法律工具幻觉评估

- **来源:** Stanford HAI, 2024
- **URL:** https://hai.stanford.edu/news/ai-trial-legal-models-hallucinate-1-out-6-or-more-benchmarking-queries
- **核心贡献:** 首个针对商用法律RAG工具的预注册实证评估
- **关键发现:**
  - Lexis+ AI、Westlaw AI-Assisted Research等商用工具仍有**17%~33%的查询产生幻觉**
  - 存在"引用真实但主张错误"的隐蔽幻觉
  - 到2026年初，已有超过700个AI幻觉案例被记录在法律程序中
  - 提出"正确性+依据性"二维幻觉分类框架

#### [12] LegalHalBench -- 法律幻觉评估基准

- **来源:** arXiv, 2025
- **URL:** https://arxiv.org/html/2501.06521v1
- **GitHub:** https://github.com/YinghaoHu/LegalHalBench
- **核心贡献:** 首个专门评估法律QA幻觉的基准
- **关键发现:**
  - 覆盖民法、刑法等7个领域
  - 提出三项自动指标：非幻觉法规率(NHSR)、法规相关率、法律主张真实性
  - 通过SFT+HIPO训练，GLM4-9B的NHSR从6.5%提升到38.4%，但仍远低于实用要求

### 3.3 检索层确定性方案

#### [13] regulated-rag -- 监管领域确定性引用RAG框架（开源）

- **来源:** GitHub开源项目
- **URL:** https://github.com/RZ-Logic/regulated-rag
- **核心贡献:** 在代码层面强制执行引用校验，而非依赖提示词
- **关键发现:**
  - 核心架构：检索后生成，生成后用**确定性校验器**检查每个引用的chunk_id是否在检索结果集合中
  - 检索置信度低于阈值时直接返回类型化拒绝，**不调用LLM**
  - 设计思路直接适用于法律领域
- **对我们的启示:** 这是最直接的解决方案——用代码级规则替代CitationAgent的LLM判断。我们已经有node_id，检查答案中的引用是否在evidence列表中是确定性的操作

#### [14] WhyHow.AI 规则化向量检索（开源）

- **来源:** WhyHow.AI / Medium
- **URL:** https://medium.com/enterprise-rag/open-sourcing-rule-based-retrieval-677946260973
- **核心贡献:** 基于规则的向量检索方案
- **关键发现:**
  - 允许开发者定义规则映射到特定文档块（如文件名+页码）
  - 通过关键词自动触发过滤器，**无需LLM参与检索路由**
  - 支持 keyword_trigger 自动匹配和 include_all_rules 多规则并行检索
- **对我们的启示:** 可以建立"备降"→CCAR-121.637/641/642/643/655的规则映射表，替代RewriteAgent的LLM改写

#### [15] Deterministic Utility Gating (MUE/DUE框架)

- **来源:** HuggingFace Papers, 2026
- **URL:** https://huggingface.co/papers/2603.18011
- **核心贡献:** 完全确定性、无需训练的证据选择框架
- **关键发现:**
  - 每个候选单元独立评估（语义相关性、词项覆盖、概念显著性、冗余性）
  - 仅当证据单元**显式陈述了任务所需信息时才接受**
  - 无满足条件的证据时系统返回"无答案"
  - **核心洞察：相似度不等于可用性**
- **对我们的启示:** 我们的top-8截断是基于RRF分数排序的，分数高不等于和问题相关。MUE的门控思路可以在检索后加一层确定性过滤

#### [16] CAR: 基于聚类的自适应检索（Coinbase生产系统）

- **来源:** arXiv, 2025
- **URL:** https://arxiv.org/pdf/2511.14769
- **核心贡献:** 用聚类分析替代固定top-k
- **关键发现:**
  - 自动检测相似度距离中的"断崖点"（紧密聚类到松散候选的转折），动态确定截断位置
  - Coinbase生产环境验证：LLM token消耗减少60%，端到端延迟降低22%，幻觉减少10%，用户参与度提升200%
  - **彻底消除了top-k参数的不确定性**
- **对我们的启示:** 我们的固定top-8导致每轮边缘法条不同，换成聚类截断可以自动确定"哪些法条真正相关"

#### [17] GroundedCache: 证据验证的安全缓存路由

- **来源:** arXiv, 2026
- **URL:** https://arxiv.org/html/2605.27494
- **核心贡献:** 四重门控机制的安全缓存
- **关键发现:**
  - 四重门控：查询相似度、检索证据重叠度、源版本有效性、词素支持度
  - HotpotQA上不安全服务率降至0%（对比朴素语义缓存的15-35%）
  - **检索侧缓存（Proximity/QVCache）比输出侧缓存（GPTCache）安全得多**
  - 词素支持门控是安全性的核心承载机制

#### [18] 混合检索融合函数分析（TM2C2 vs RRF）

- **来源:** arXiv, 2022
- **URL:** https://arxiv.org/pdf/2210.11934
- **核心贡献:** 系统分析凸组合和RRF两种融合策略
- **关键发现:**
  - 凸组合对归一化方式不敏感，但需归一化
  - RRF基于排名丢弃分数分布信息，跨域泛化差
  - **凸组合仅需少量标注查询即可调优alpha参数，样本效率极高**
  - TM2C2（理论最小值归一化+凸组合）在不同数据分布下最稳定
- **对我们的启示:** 我们的RRF融合（BM25 0.35 + 向量 0.35 + hint 0.30）可以考虑换成凸组合，法律领域分布稳定，适合参数化融合

#### [19] Stable-RAG: 消除检索排列导致的幻觉

- **来源:** arXiv, 2026
- **URL:** https://arxiv.org/html/2601.02993v4
- **核心贡献:** 揭示了文档排列顺序对LLM答案的影响
- **关键发现:**
  - 相同文档集仅因排列顺序不同，LLM答案就会大幅变化
  - 通过多次排列下的隐状态聚类 + DPO对齐，使模型对文档顺序不敏感
  - **检索结果的排列顺序也是不确定性的来源之一**
- **对我们的启示:** 即使检索结果完全相同，传递给SynthesisAgent的证据排列不同也会导致不同答案。可以对证据按法律层级排序（上位法→下位法）消除排列随机性

#### [20] 查询重写的启发式流水线（无LLM方案）

- **来源:** GitHub (rag-from-scratch)
- **URL:** https://github.com/pgusto/rag-from-scratch/blob/main/examples/06_retrieval_strategies/05_query_rewriting/CONCEPT.md
- **核心贡献:** 完整的无LLM查询重写流水线
- **关键发现:**
  - 规范化 → 意图分类 → 指代消解 → 安全过滤 → 重写 → 结构化约束 → 质量门控
  - 启发式包括：去除填充词、缩写映射表、注入模式剥离、比较查询拆分
  - **启发式优先、LLM可选升级、失败回退启发式**
- **对我们的启示:** 这是最务实的方案——法律领域术语有限（备降、签派放行、运行规范、CCAR-121等），完全可以建立术语→法条的映射表

#### [21] "智能"查询重写何时破坏RAG

- **来源:** Medium (Nexumo)
- **URL:** https://medium.com/@Nexumo_/when-smart-query-rewriting-breaks-your-rag-494271e751d3
- **核心贡献:** 分析了10种LLM查询改写导致RAG检索失败的案例
- **关键发现:**
  - 规模化场景下LLM改写引入的不确定性远超其带来的收益
  - 解决方案：在调用LLM改写前增加**确定性预检查**（如tokenize后与历史问题比较），避免不必要的LLM调用

#### [22] GPTCache: 开源语义缓存

- **来源:** Zilliz, GitHub
- **URL:** https://github.com/zilliztech/gptcache
- **核心贡献:** 最成熟的开源LLM语义缓存方案
- **关键发现:**
  - 将查询转embedding后做向量相似搜索匹配历史缓存
  - 支持多种向量库（Milvus/FAISS/Chroma）和嵌入模型
  - 但GroundedCache论文指出其朴素使用时存在86%的响应劫持风险，需要额外安全门控

## 四、方案对照分析

### 4.1 我们系统的架构 vs 学术界的建议

| 我们的环节 | 当前实现 | 学术界建议 | 可落地方案 |
|-----------|---------|-----------|-----------|
| SubjectAgent | LLM提取主体 | 可保留（不直接影响检索） | — |
| IssueAgent | LLM识别争点 | 可保留（不直接影响检索） | — |
| **RewriteAgent** | **LLM改写queries** | **去LLM化，规则映射** | **关键词→法条映射表** |
| **RetrievalAgent** | **RRF融合 + top-8截断** | **凸组合 + 聚类截断** | **TM2C2 + CAR** |
| CrossRef | 规则补全 | 可保留 | — |
| **CitationAgent** | **LLM判断校验状态** | **代码级集合成员检查** | **regulated-rag模式** |
| ConflictAgent | 规则层级排序 | 可保留 | — |
| SynthesisAgent | LLM生成答案 | 加强格式约束 | few-shot示例 + 后处理 |
| **ReflexionAgent** | **LLM自检(从未通过)** | **缓存复用或移除** | **检索结果缓存** |

### 4.2 按优先级排序的改进路线

#### P0 — 消除主要不确定性（预期消除70%+的波动）

**P0-1: 检索去LLM化 — 关键词规则映射**

来源：WhyHow.AI, 启发式流水线, "智能查询重写何时破坏RAG"

思路：建立法律术语→法条的确定性映射表，绕过RewriteAgent的LLM改写。

```
"备降" → CCAR-121.637, 121.641, 121.642, 121.643, 121.655, 121.663
"运行规范" → CCAR-121.20, 121.21, 121.117
"签派放行" → CCAR-121.631, 121.633, 121.635
```

工程量：中（需要构建映射表，但法律领域术语有限）

**P0-2: 引用校验去LLM化 — 代码级集合成员检查**

来源：regulated-rag

思路：CitationAgent的LLM判断换成确定性检查——答案中引用的法条是否在检索到的evidence集合中。

```
# 伪代码
for citation in answer_citations:
    if citation.node_id not in {e.node_id for e in evidence}:
        citation.status = "unsupported"  # 确定性，无LLM参与
```

工程量：小（已有node_id，检查集合成员是O(1)操作）

#### P1 — 减少边界不确定性（预期再消除15%的波动）

**P1-1: 自适应top-k — 聚类截断**

来源：CAR (Coinbase生产系统)

思路：不用固定top-8，而是检测RRF分数的"断崖点"，自然截断。

工程量：中（需要实现聚类算法，但Coinbase已验证生产可行性）

**P1-2: RRF→凸组合（TM2C2）**

来源：TM2C2论文

思路：法律领域分布稳定，凸组合的alpha参数可以固定，比RRF更稳定。

工程量：小（改融合函数，参数可在少量标注数据上调优）

#### P2 — 缓存与格式优化（预期再消除10%的波动）

**P2-1: 检索结果缓存**

来源：GroundedCache

思路：相同或语义相似的查询直接复用检索结果，跳过整个检索和自检。

工程量：小（已有向量模型，加一层缓存即可）

**P2-2: 答案格式约束**

来源：Ark Legal, 4xxi三层验证

思路：在SynthesisAgent的prompt中用few-shot示例规定法条引用格式，或后处理提取+校验。

工程量：小

#### P3 — 长期优化（需要训练能力）

**P3-1: PS-GRPO强化学习训练一致性**

来源：Con-RAG

思路：用同一查询的多个改写版本生成多组输出，用组间相似度作为奖励信号训练。

工程量：很大（需要训练基础设施）

## 五、关键论文的技术细节

### 5.1 ReproRAG的不确定性量化指标

```
Exact Match Rate (EMR): 检索结果与参考结果的完全匹配率
Jaccard Similarity: 检索结果集合的重叠度
Kendall's Tau: 排序相关性
```

这三个指标可以直接用于量化我们检索层的一致性。我们10轮的EMR=0%（没有两轮完全相同），Jaccard均值约40%（68种法条中平均每轮12种）。

### 5.2 Con-RAG的三维一致性分解

```
检索层一致性: 同一查询多次检索结果的稳定性
生成层一致性: 相同检索结果下LLM输出的稳定性
端到端一致性: 完整管线的输出稳定性
```

我们的情况：
- 检索层：0分（10轮10种组合）
- 生成层：低分（相同证据下校验状态矛盾）
- 端到端：60分（结论严格一致率）

### 5.3 Deterministic RAG的六级不确定性模型

```
Layer 1: Embedding (向量模型输出)
Layer 2: Index Algorithm (HNSW/IVF等近似最近邻)
Layer 3: Ranking Score (浮点tie-breaking)
Layer 4: Context Assembly (上下文组装)
Layer 5: LLM Sampling (LLM采样)
Layer 6: Cache Behavior (缓存行为)
```

我们的不确定性主要在 Layer 4（LLM改写queries导致不同的上下文组装）和 Layer 5（LLM生成和校验的不确定性）。

### 5.4 GroundedCache的四重门控

```
Gate 1: 查询相似度 (query embedding距离)
Gate 2: 检索证据重叠度 (evidence set overlap)
Gate 3: 源版本有效性 (source version check)
Gate 4: 词素支持度 (lexical support score)
```

这四个门控确保缓存复用的安全性，可以直接用于我们的检索结果缓存。

## 六、结论

### 6.1 核心判断

我们系统的一致性问题不是实现质量问题，而是**架构性问题**。当前的LLM查询改写→检索→LLM校验→LLM生成→LLM自检的流水线，把5个非确定性环节串联在一起，任何一个环节的波动都会级联放大。

学术界和工业界已经形成了共识：

1. **检索路由不应该依赖LLM** — 启发式规则 > LLM改写
2. **引用校验不应该依赖LLM** — 代码级集合检查 > LLM判断
3. **RAG不能根除幻觉** — 需要确定性的后验证机制
4. **temperature=0不等于确定性** — API服务端仍有非确定性
5. **固定top-k应该被替代** — 聚类截断 > 固定值

### 6.2 推荐的改进路径

**第一步（P0, 消除70%波动）:**
- 把 RewriteAgent 的 LLM 改写换成确定性的关键词→法条映射表
- 把 CitationAgent 的 LLM 校验换成确定性的集合成员检查

**第二步（P1, 再消除15%波动）:**
- 把固定 top-8 换成聚类自适应截断
- 把 RRF 融合换成凸组合（TM2C2）

**第三步（P2, 再消除10%波动）:**
- 加检索结果缓存
- 加强答案格式约束

完成这三步后，预计端到端一致性可以从当前的60%提升到95%+。
