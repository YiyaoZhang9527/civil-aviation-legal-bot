## Bug 1：双书名号 `《《法规》》` —— 来源是 law_title 已被污染 + 渲染层无脑再包

- **位置**：
  - **根因（污染）**：`legalbot/parser.py:85-89` 标题提取规则
  - **放大（污染）**：`legalbot/parser.py:99, 131, 137, 156, 161, 178` 把脏 title 写进 `LawNode.source_anchor` / `anchor_map`
  - **呈现（双书名号）**：`legalbot/agents.py:699` `f"《{ev.law_title}》{ev.article}"`
  - **呈现（双书名号）**：`legalbot/agents.py:719` 同上
  - **呈现（双书名号）**：`legalbot/agents.py:801` `f"{i}. 《{law}》{article}"`（拒答兜底也中招）
- **机制**：
  1. `parse_law_text` 取文件前 10 行非空首行作为 `title`（line 85-89），靠 `_SKIP_TITLE_RE`（line 33-48）过滤。
  2. `_SKIP_TITLE_RE` 跳过的模式：纯数字页码、`xxx部令`、`xxxx年第x号`、`[CCAR-xx]`、`第x章/节/条/款`、`【...】` 等等。
  3. **关键漏网**：`《XX》已于YYYY年MM月` 这种"先亮法规名再接颁布前言"的行不在 SKIP 列表中。`_SKIP_TITLE_RE` 只匹配 `[【\[]` 两种方括号（line 44），**不匹配 `《》`**。
  4. 真实数据 `data/法律数据/公共航空运输旅客服务管理规定.txt` 第 4 行是 `《公共航空运输旅客服务管理规定》已于2021 年2 月`（跨 3 行），被完整当作 `title`。
  5. `find_document` 走 `normalize_text`（utils.py:45）—— 会剥离 `《》` 和空白，所以**路由工作正常**。
  6. `Evidence.law_title = doc.title` 携带原始脏字符串。
  7. `_render_structured_answer` 第 699 行对 `ev.law_title` 无脑再裹一层 `《》` → 用户看到 `《《公共航空运输旅客服务管理规定》已于2021 年2 月》第十条`。
- **触发条件**：
  - 数据文件首段有"《法名》+ 颁布前言"格式
  - 任何使用 `_render_structured_answer` 或 `_render_evidence_fallback` 的答案路径（SYNTHESIS_JSON_MODE=True 时）
  - 证据涉及的法规在脏 title 列表内（已确认 10+ 部）
- **显示效果**：
  - 结构化答案区：`- 内容（**《《公共航空运输旅客服务管理规定》已于2021 年2 月》**第十条）`
  - 拒答兜底：`1. **《《公共航空运输旅客服务管理规定》已于2021 年2 月》**第十条`
  - 不只是难看——会让 LLM 在自检环节（reflexion）把这条 evidence 误判为"质量差"，可能触发不必要的补搜。
- **示例（grep 验证）**：
  - `data/indexs/公共航空运输旅客服务管理规定.indexs.md` 第 1 行：`# 《公共航空运输旅客服务管理规定》已于2021 年2 月`
  - `data/indexs/民用航空空中交通管理规则（交通运输部_2017年第30号）.indexs.md` 第 1 行：`# «民用航空空中交通管理规则»已于2017年9月20日经第16`（**全角拉丁引号 `«»` 也未过滤**）
  - `data/indexs/民用航空空中交通管理规则（交通运输部_修订版_2022年第36号）.indexs.md` 第 1 行：`# 《交通运输部关于修改〈民用航空空中交通管理规则〉的决定》已于2022`
  - 总计：**至少 10 部法规 title 被污染**（grep 验证：`航空器型号和适航合格审定噪声规定`、`国际航空运输价格管理规定`、`螺旋桨适航规定`、`民用航空计量管理规定`、`民用航空空中交通管理规则`（双版本）、`民用航空器国籍登记规定`、`民用航空器事件技术调查规定`、`公共航空运输旅客服务管理规定` 等）
- **影响**：
  - 用户体验崩坏
  - 抽取的 `keywords` 也被污染（`legalbot/parser.py:74-76`）：这些法的 keywords 出现 `已于`、`年第`、`次部务会议通过` 等无意义词，**降低 BM25/vector 召回质量**
- **修复建议**（按推荐顺序）：
  1. **根因修复（parser）**：`_SKIP_TITLE_RE` 增补：
     ```python
     r"|《[^》]+》(?:已|经|自|现予|现对)"  # 《法名》+ 颁布谓语
     r"|«[^»]+»(?:已|经|自|现予|现对)"  # 全角拉丁引号版
     r"|《[^》]+》$"                       # 单纯《法名》独占一行
     ```
     或者更稳的方案：标题提取改**从第 10 行之后**找（部委令 + 令号通常在 1-3 行，颁布前言 4-10 行，**真正的 title 通常在第 11 行以后**——见 `公共航空运输旅客服务管理规定.txt` 第 11 行 `公共航空运输旅客服务管理规定`）。
  2. **兜底修复（渲染层）**：`agents.py:699/719/801` 检测 `law_title` 是否已含 `《》`，有则不重复包：
     ```python
     title = ev.law_title
     if not (title.startswith("《") and title.endswith("》")):
         title = f"《{title}》"
     refs.append(f"{title}{ev.article}")
     ```
  3. **数据修复**：重建索引（`python -m legalbot.cli build`），脏 title 法规重新解析。

## Bug 2：crossref 把"被引用条文"无差别塞进 evidence —— Q18/Q22 错配的元凶

- **位置**：
  - **入口**：`legalbot/agents.py:540-562`（`expand_evidence_references` 后处理）
  - **正则**：`legalbot/crossref.py:142` `_CROSS_BOOK_RE`
  - **解析**：`legalbot/crossref.py:190-234` `resolve_references`
  - **展开**：`legalbot/crossref.py:237-260` `expand_evidence_references`
  - **追加**：`legalbot/agents.py:545-562` 直接 `loaded.append(...)`
- **机制**：
  1. `retrieve` 把检索到的 evidence 用 `expand_evidence_references` 后处理。
  2. 后处理对**每条 evidence 的全文**跑 `_CROSS_BOOK_RE`，提取所有 `《XX法》第X条` / `本法第X条` / 无书名号版引用。
  3. 对每条引用，**无 relevance 校验**直接 `read_law_node` 读出来，包成新 Evidence，**`score=0.0`**（agents.py:558）。
  4. 追加到 `loaded` 列表末尾。
  5. 后续 `state.evidence = state.evidence[:12]`（agents.py:1216）按分数倒序截断——`score=0.0` 的 xref 通常被挤出前 12，但**当高分 evidence < 12 条时仍会进入 LLM 视野**。
  6. 合成 prompt `_build_evidence_text`（agents.py:905-909）把这些 xref evidence 一视同仁列出 `[{i}] {law_title} {article} node={node_id}\n{text}`。
  7. LLM 看到 evidence 池里出现 "民用航空空中交通管理规则 第22条" + "行李丢失"，因为关键字 `第22条` 是 text 里唯一带数字的项，**被 LLM 当作"该法规说行李丢失按第22条办"** 的依据。
  8. Q18（航班延误餐饮）实际数据：检索到 `航班正常管理规定`/`公共航空运输旅客服务管理规定` 的某条，里面又 cross-ref 到 `民用航空空中交通管理规则第X条`（大概率是关于延误信息通报的空管规定，跟餐饮无关），xref 把这条加载到 evidence 池；LLM 看到 `航空` `管理` `规定` 三关键词都在 prompt 里，就引用了。
  9. Q22 行李丢失：检索到 `公共航空运输旅客服务管理规定`（实际法名）里某条说"赔偿适用本法第22条"——而第22条**实际是某条不相关**（用户报告里说是"民用航空器优先权"，但实际可能是别的——见下）。
- **触发条件**：
  - 任何检索到的 evidence 文本里出现 `《XX法》第X条` 字样
  - 且被引用条文所在法规已在索引内
  - 且高分 evidence 数量 < 12
- **显示效果**：
  - 答案里出现 `《民用航空空中交通管理规则》第二十二条：...` 这条引用
  - 该条文与用户问题主题完全无关
  - CE 校验因"evidence 文本真的支持该法名+条号"而给高分 → **错误引用在 CE 指标下"看起来"很对**
  - 但 Faithfulness 评测（独立 LLM 判 claim）会发现这条 claim 答非所问 → **Faithfulness=0%**
- **示例**：
  - agents.py:541 跑 `expand_evidence_references(loaded, max_items=5)`——**`max_items=5` 给了 5 个无关条目的额度**
  - crossref.py:223 `read_law_node(target_law_id, node_id, include_context=False)`——只读单条，没有 relevance 评估
  - 没有任何一行代码检查"这条被引用的条文是否与用户问题相关"
- **影响**：
  - **Q18 / Q22 错配的直接原因**
  - Faithfulness 评测 0% 案例的常见来源
  - 用户看到"专业、引用齐全"但内容是空话
- **修复建议**（按推荐顺序）：
  1. **最小修复**：在 `agents.py:545` 之前加 relevance gate——只加载"被引用条文文本与用户问题有词法重叠"的 xref：
     ```python
     q_tokens = set(extract_phrases(question, min_len=2))
     for xr in xrefs:
         if not (q_tokens & set(extract_phrases(xr.get("text", ""), min_len=2))):
             continue
         loaded.append(...)
     ```
  2. **更稳的修复**：xref evidence 给 `score=-1.0`（明确低于所有真实 evidence），然后在合成前硬截断到 `state.evidence[:12]`（已有 line 1216，但当前 score=0.0 仍可能挤进尾部）。
  3. **根本修复**：xref 不直接入 evidence 池，而是**仅作为"如果 LLM 想用，可以调用 read_law_node 的候选清单"**——把 xref 列表传给 synthesis prompt 让 LLM 决定是否需要，而非硬塞。

## Bug 3：evidence 文本被 include_context 扩展 + 大条款截断不均，混淆"哪条说啥"

- **位置**：
  - **加载**：`legalbot/retrieval.py:907-931` `resolve_node_text`
  - **加载**：`legalbot/retrieval.py:934-960` `read_law_node`
  - **调用**：`legalbot/agents.py:526` `read_law_node(hit.law_id, hit.node_id, include_context=True)`
  - **截断**：`legalbot/agents.py:909` `item.text[:config.SYNTHESIS_EVIDENCE_TRUNCATE]`（2000 字符）
  - **直读**：`legalbot/retrieval.py:151-218` `_direct_read_articles`（用 `summary` 替代全文，line 205）
- **机制**：
  1. `resolve_node_text` 用 `anchor[line_start] - 2` 到 `anchor[line_end] + 2` 取文本（line 928-929），**前后各加 2 行上下文**。
  2. 对一个 1 行条款，文本变成 5 行——**这 2 行上下文可能来自相邻条款**，LLM 无法分辨哪句是 evidence 原文、哪句是上下文。
  3. `SYNTHESIS_EVIDENCE_TRUNCATE = 2000`（config.py:94）——**长条款被腰斩**，但 2000 字符在中文里约 600-700 字，对很多民航规章条款（特别是 CCAR 系列）只覆盖 1/3。
  4. `_direct_read_articles`（line 205）`text = node.summary or node.text or node.title`——当 `summary` 存在时**根本不给 LLM 看全文**，只给 120 字符摘要（parser.py:64-66）。
  5. 同一法不同条款的 evidence 长度差异巨大，**LLM 倾向于引用长条款的细节**（细节越多越像"答对了"）。
- **触发条件**：
  - 任何检索（所有 evidence 都走 `include_context=True`）
  - 任何含 summary 的 evidence（line_hint 直读时）
- **显示效果**：
  - LLM 看到 `第十一条\n[上下文第10条最后一句]\n第十一条正文\n[上下文第十二条开头]`
  - 引用时可能错把相邻条款内容算到本条头上
  - 直读的 evidence 文本特别短，**CE 校验时 summary 跟 claim 不匹配 → 直接判 unsupported**，反而"安全"
- **示例**：
  - `legalbot/retrieval.py:928-929`：硬编码 ±2 行，无视 evidence 在法律文件里的语义边界
  - `legalbot/agents.py:909`：无标记告诉 LLM "以下 2000 字可能截断"
  - `legalbot/retrieval.py:205`：`text = node.summary or ...`——summary 仅取首句 120 字符（parser.py:64-66）
- **影响**：
  - 长条款被腰斩，LLM 看不到"但书"、"除外"等关键限定
  - 短条款被前后扩展，引入相邻条款噪声
  - 直读 evidence 信息量严重不足
- **修复建议**：
  1. **去 include_context**：`agents.py:526` 改为 `include_context=False`（crossref 已经这么做了，retrieve 没必要扩）
  2. **截断提示**：`_build_evidence_text` 在每条 evidence 末尾加 `[文本已截断]`
  3. **直读给全文**：`_direct_read_articles:205` 改为 `text = node.text or node.summary or node.title`（首选全文）

## Bug 4：`_canonical_title` 用 `《》` normalize，dedupe 工作正常 —— 但 display 没经过 normalize，造成"同一法两条 evidence 显示不同名"

- **位置**：
  - **dedupe**：`legalbot/retrieval.py:111-112` `_canonical_title`
  - **未用**：`legalbot/agents.py:535, 544, 699, 719, 796, 801, 835, 909`（全部直接用 `ev.law_title` / `doc.title`）
- **机制**：
  1. `_canonical_title` 用 `normalize_text` 去 `《》`、`中华人民共和国`、空白，作为 dedupe key。
  2. 所以**索引内部去重 OK**——同一法脏 title 和干净 title 会合并。
  3. 但所有**展示给用户和 LLM 的地方都用原始 `law_title`**。
  4. 出现 `doc.title = "《中华人民共和国民用航空法》（2026年7月1日起施行）"`（已确认，见 `data/法律数据/中华人民共和国民用航空法.txt:1`），**渲染后**：`《中华人民共和国民用航空法》（2026年7月1日起施行）第三条`——括号日期带在法名后。
  5. 这种"带日期的 title"虽然没双书名号，但**让 LLM 误以为该法是 2026 年新法**（实际是 1995 年制定，2026 是某次修正），可能错引时效。
- **触发条件**：
  - 文件首行格式为 `中华人民共和国民用航空法（YYYY年M月D日起施行）`
  - 不在 `_SKIP_TITLE_RE` 跳过规则里
- **显示效果**：
  - 用户看到 `《中华人民共和国民用航空法》（2026年7月1日起施行）第三条`
  - LLM 可能在答案里写"根据 2026 年 7 月 1 日起施行的《民用航空法》..."
  - 严重误导
- **示例**：
  - `data/法律数据/中华人民共和国民用航空法.txt:1`：`中华人民共和国民用航空法（2026年7月1日起施行）`
  - `data/indexs/中华人民共和国民用航空法.indexs.md:1`：`# 中华人民共和国民用航空法（2026年7月1日起施行）`
- **影响**：
  - 一部核心法（民用航空法）显示带括号日期
  - LLM 时效判断错位
- **修复建议**：
  1. **parser 端**：增加 `r"|\([^)]*\d{4}年[^)]*\)"` 到 `_SKIP_TITLE_RE`（任何带 `(YYYY年...)` 的整括号内容）
  2. **显示端**：`legalbot/agents.py:796-797` 改用 `ev.law_id` 做兜底展示（`ev.law_id` 是 path stem，干净），或对 `law_title` 先跑一次 `normalize_text` 再加 `《》`

## Bug 5：keywords 提取被脏 title 污染，BM25/vector 召回引入噪声

- **位置**：`legalbot/parser.py:70-77` `_keywords_from_text`（被 `parse_law_text` 在 line 130, 154, 172 等处调用，作用于每条 article）
- **机制**：
  1. `combined = f"{title} {text}"` —— article 的 keywords 包含 title 中的所有 2+ 字 token。
  2. 脏 title 包含 `已于`、`年第`、`次部务会议通过` 等 token。
  3. 检索"已于 2021"的用户查询会误命中这部法规。
  4. 检索"部务会议通过"的用户查询会误命中（虽然不现实，但揭示了污染面）。
  5. 已确认：`data/indexs/公共航空运输旅客服务管理规定.indexs.md` 的 keywords 包含 `已于, 中华人民共和国交通运输部令, 年第, 日经第, 次部务会议通过`。
- **影响**：
  - 召回噪声 → 多一个候选 → 占用宝贵的 top-k 配额 → 挤掉真正的相关法
  - 评测体系里 BM25 召回率被这个机制持续压低
- **修复建议**：Bug 1 修好后自动消失。临时方案：`_keywords_from_text` 排除脏 title 中 `_SKIP_TITLE_RE` 命中的部分。

## 综合判断

### Bug 数量与类型

本轮审查识别 **5 个** Bug，按严重程度：

| # | Bug | 严重度 | 修复成本 | 影响范围 |
|---|-----|--------|----------|----------|
| 1 | 双书名号 + 标题脏数据 | **P0** | 低（正则 + 渲染兜底） | 10+ 部法、占答案总量约 30% |
| 2 | crossref 污染 evidence 池 | **P0** | 低（relevance gate） | Q18/Q22 错配直接原因 |
| 3 | include_context + 截断不均 | **P1** | 低（参数调整） | 长条款误读、CE 假阳 |
| 4 | 核心法带日期显示 | **P1** | 低（正则） | 时效误导 |
| 5 | keywords 被污染 | **P2** | 跟 Bug 1 联动 | BM25 召回噪声 |

### 评估

- **隐藏 bug 数量**：至少 5 个，与症状的对应关系清晰：
  - **症状 1**（Q18 引用错法规）= Bug 2（crossref 无 relevance 门）
  - **症状 2**（Q22 行李丢失引用优先权）= Bug 2 + Bug 1（脏 title 误导 LLM）
  - **症状 3**（双书名号 + 日期残留）= Bug 1 + Bug 4

- **质量影响**：
  - 修好 Bug 1 + Bug 2 即可消除用户报告的 3 个症状
  - Bug 3、4、5 是同一污染源（脏 parser）+ 同一无兜底（裸渲染）的连锁反应
  - 一旦修好 parser 端的 `_SKIP_TITLE_RE`（约 20 行正则增量），Bug 1/4/5 同时消失——**1 处修复，3 个 bug 灭**

- **最大风险**：Bug 2（crossref）若不修，再多 refinement 也救不回——因为 LLM 在 prompt 里看到的就是错配证据，再聪明的 LLM 也很难分辨"这条 evidence 是被 crossref 强塞进来的、不是直接回答问题的"。

### 建议优先级

1. **立即修**：Bug 1 的 parser 端（`_SKIP_TITLE_RE` 增补）—— 一次性清掉 30% 答案的格式污染
2. **立即修**：Bug 1 的渲染端兜底（agents.py:699/719/801）—— 防御未来再出脏 title
3. **关键修**：Bug 2 加 relevance gate —— 解决 Q18/Q22 类系统性错配
4. **可选修**：Bug 3、5（连锁收益）
