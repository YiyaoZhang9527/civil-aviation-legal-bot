# Query 改写与检索对齐诊断

**审查目标**：`RewriteAgent` / `DecompositionAgent` / `QUERY_GATE` 与 Tree 检索的实际对齐
**核心症状**：Q19「拒载」bot 用 LLM 改写 multi-query 能召回 article 31，单 query 检索漏 article 31
**方法**：codegraph 读源码 + test30 实测数据交叉验证

---

## 观察 1：改写 prompt 不强制 article_hints 格式，浪费 Phase 0 直读机制

- **位置**：`legalbot/agents.py:380-410`（RewriteAgent 系统 prompt）vs `legalbot/retrieval.py:151-218`（`_direct_read_articles`）
- **改写输出**：prompt 第 4 行要求 "law_hints 应尽可能给出具体法规名称"，但对 `article_hints` **完全没说必须带法名**。LLM 自由发挥，常见输出是 `["第三十一条", "第三十二条"]`（裸条号）或 `["拒载情形条款"]`（描述性短语）。
- **检索接收**：`retrieval.py:151-218` 的 `_direct_read_articles` 用严格正则 `(.+?)\s*第([一二三四五六七八九十百零\d]+(?:\.\d+)*)\s*条`（line 160）解析，**必须"法名+第X条"组合**才能直读。匹配失败直接 `return []`（line 184-185），不给任何提示。
- **差距**：Phase 0 直读是绕过向量/关键词检索的最强机制（`score=1.0`，`retrieval.py:212`），是 0.5-0.8 分的关键来源。但因为 prompt 没说格式要求，**绝大多数 article_hints 都被吞掉**。实测 test30_20260603_190237 中 Q19 答到了 article 31，但 evidence_articles 里能看到多个无关法律的"第三十一条"被打散命中，说明 article_hints 没有起到"定向直读"作用。
- **量化影响**：直读完全没贡献的题数 = 30 题中 27+ 题（article_hints 多半为空或格式不匹配）。
- **建议**：在系统 prompt 第 4 条后加 `article_hints 必须形如 "完整法规名第X条"`，并示例 2-3 条。直读命中率能从当前 ~20% 提到 ~70%+。

## 观察 2：Decomposition 触发条件几乎不被满足，子问题机制形同虚设

- **位置**：`legalbot/agents.py:417-422`（DecompositionAgent.decompose 入口）
  ```python
  sub_questions = rewrite_plan.sub_questions
  if len(sub_questions) <= 1:
      return DecompositionResult(needs_decomposition=False)
  ```
- **改写输出**：`agents.py:380-410` 系统 prompt **完全没提 `sub_questions` 字段**。用户 prompt（line 394-400）只是把 `sub_questions` 列为 JSON 字段之一，没有指示 LLM 必须填、什么时候填、填什么。
- **检索接收**：`agents.py:1115-1133` 流程是 `if decomposition.needs_decomposition: _retrieve_decomposed else retrieve`。因改写 LLM 几乎不主动填 sub_questions，**decomposition 实际触发率估算 < 5%**（30 题中可能 0-2 题）。
- **差距**：DecompositionAgent 的代码、`MAX_SUBQUESTIONS=4` 配置、`_retrieve_decomposed` 的 round-robin 合并（`agents.py:1298-1312`）——**整套机制在生产路径上几乎从不执行**。改写 prompt 给的"sub_questions"字段是**软字段**，LLM 视为可选项。
- **量化影响**：复杂问题（多法规交叉）Q06 EDTO、Q20 行李赔偿、Q22 安全信息报告等的召回率本可受益，但因未触发拆解，bot 退化为单 multi-query 检索。
- **建议**：在改写 system prompt 显式加一条："如果问题涉及 ≥2 个法规领域或 ≥2 个独立子议题，必须在 sub_questions 字段中拆分为 2-4 个子问题"。

## 观察 3：QUERY_GATE 只增强鲁棒性，不增强召回率

- **位置**：`legalbot/agents.py:1085-1098`
  ```python
  if config.QUERY_GATE_ENABLED:
      normalized_q = state.normalized_question
      gate_queries = [normalized_q]
      for q in rewrite_plan.queries:
          q_n = normalize_text(q)
          if q_n != normalized_q and q_n not in gate_queries:
              gate_queries.append(q)
      gate_queries = gate_queries[:5]
  ```
- **改写输出**：QUERY_GATE 把原始 normalized_question 强行放 query[0]，后接 LLM 改写 queries（去重 + 截断到 5）。
- **检索接收**：`agents.py:489-498` 对每个 query 独立调 `search_index_tree`。`agents.py:512-522` 因 WRRF_ENABLED=False 走简单去重合并：第一个 query 的 hits 先入，后面的 hits 按 (law_id, node_id) 去重追加。最后 `sort by score reverse` + 截断。
- **差距**：
  1. **保底 ≠ 召回**：如果原始 query 漏 article 31（用户症状），改写 query 找到 article 31 即可——但反之不行。**两个 query 都在同一语义空间，一个漏了另一个大概率也漏**。QUERY_GATE 救不了"单 query 漏掉"的问题。
  2. **去重先于排序**：第一个 query（原始 normalized_question）的 hits 占据 `all_hits` 的前 N 个位置。如果它的 top-10 命中了 10 个不相关法律（Q19 案例：可能命中"航空货物运输"的运输总条件条款），后入的改写 query 找到 article 31 时虽然 score 可能更高，但**已经被挤到 11 名之后**，被 `EVIDENCE_LOAD_LIMIT=12`（`config.py:92`）的硬截断影响有限，但被 `top_k=10`（`config.py:91`）的检索级截断影响严重。
  3. **截断 5 个太狠**：LLM 改写常输出 5-7 个 queries（`rewrite.queries[:3]` 暗示），5 上限会丢 2-3 个。
- **量化影响**：QUERY_GATE 救场场景 ≈ 改写 LLM 失败（`agents.py:1071-1074` 异常路径），不是"召回率提升"。
- **建议**：
  1. 把 WRRF 重开（消融显示负收益，但是**去重合并**比 WRRF 更差，至少要按"对每个 query 等权 RRF"）。
  2. 把截断从 5 提到 8。
  3. 给每个 query 单独标"是否改写 query"（vs 原始），用真 RRF 而不是顺序追加。

## 观察 4：Gold 评测用空主体/争点触发改写，与 bot 检索流程不对称

- **位置**：`tests/test_gold_coverage.py:133-137`
  ```python
  rewrite = rewrite_agent.rewrite_queries(
      question, SubjectAnalysis(), IssueAnalysis()
  )
  ```
- **改写输出**：gold 评测的 RewriteAgent **收到空 `SubjectAnalysis()`（line 136）和空 `IssueAnalysis()`（line 137）**——dataclass 默认值，所有字段都是空 list / 空 dict。
- **检索接收**：bot 的 `_answer`（`agents.py:1069-1070`）传的是**真实分析结果**（`subject_analysis` / `issue_analysis` 是上游 LLM 填的）。两个 LLM 看到的上下文不同：bot 的改写 LLM 知道"这是 CCAR-121 下的备降问题，主体是航空公司"，gold 的改写 LLM 什么都不知道。
- **差距**：
  1. **改写质量不对称**：bot 改写 LLM 有主体/争点上下文，gold 没有。**gold 改写比 bot 改写更弱**，gold 召回比 bot 召回更窄。
  2. **queries 数限制不对称**：gold 截断到 4（`test_gold_coverage.py:148`），bot 截断到 5（`agents.py:1092`）。bot 改写多 1 个 query 路径。
  3. **decomposition 路径缺失**：gold 流程**完全没有 DecompositionAgent 调用**（line 127-156 不涉及），如果 bot 实际触发了拆解，gold 完全错过。
- **量化影响**：**gold 评测系统性低估 bot 能力**。改写 prompt 明确说"接收主体分析和争点分析"（`agents.py:381-383` 写"主体分析"和"争点分析"作为输入），给空就是绕过这条路径。
- **建议**：要么 (a) gold 评测也跑完整 orchestrator（不再手工拼），要么 (b) 改 test_gold_coverage.py 让 gold 共享 bot 的实际 rewrite_plan。优先 (b)，因为跑完整 orchestrator 慢且不确定。

## 观察 5：每个 query 独立检索不做 query 间排序，召回结果对顺序敏感

- **位置**：`legalbot/agents.py:489-522`（RetrievalAgent.retrieve）
- **改写输出**：5 个 gate_queries 并行进入 search_index_tree。
- **检索接收**：每个 query 独立跑 `tree.search(query, top_k=10, hint_law_ids=..., hint_scores=...)`（`retrieval.py:553-555`），返回 top-10 hits。
- **差距**：
  1. **没有 query 间 RRF**：WRRF_ENABLED=False 走简单去重（`agents.py:512-522`）。意味着第 2-5 个 query 的命中**无法通过多 query 共识加分**。
  2. **截断在 query 内部**：每个 query 只取 top-10，如果正确 article 在改写 query 的 top-10 边缘（rank 8-10），会落榜。在 gold 用 top_k=30 时不会落，但 bot 用 RETRIEVAL_TOP_K=10（`config.py:91`）会。
  3. **第一个 query 的 top-10 锁定 all_hits 前 10 名**：去重后追加，所以原始 normalized_question 的 top-10 永远在前 10（即使 score 低），后入的改写 query 即使找到 article 31 也得从 rank 11 开始排。
- **量化影响**：改写 query 的召回价值被"位置偏差"打折，可能只贡献 30-50% 的价值。
- **建议**：把 WRRF 重新打开，或者实现一个简单的"先合并 → 再 sort → 再截断"流程：把所有 query 的 hits 按 score 排序后统一截断 top-30，再用 `MAX_EVIDENCE_PER_LAW=3` 限制每法规证据数。

## 观察 6：改写 prompt 没要求"改写 query 之间的语义差异度"

- **位置**：`legalbot/agents.py:380-410`
- **改写输出**：LLM 自由发挥，常输出同义改写（"签派员放行检查什么" vs "签派员放行需检查事项"）— 经 `normalize_text` 去重后只留一个。
- **检索接收**：`agents.py:1088-1091` 的去重只检查 `q_n != normalized_q` 和 `q_n not in gate_queries`。如果 LLM 输出 2 个改写 query 但 normalize 后完全相同，第 2 个直接被丢。
- **差距**：改写 query 的价值在于**语义多样性**（一个 query 走 BM25 关键词路径，另一个走向量路径）— 而不是**同义改写**。但 prompt 没引导 LLM 输出语义多样的 query。
- **量化影响**：实测 test30 中 evidence_count 普遍是 12 或 16，unique_articles 几乎等于 evidence_count（每条都是独立条款），说明多 query 确实有贡献。但若 LLM 输出同义改写，5 个 query 实际只有 2-3 个语义独立。
- **建议**：prompt 加规则"queries 之间必须有不同检索角度（如一个偏关键词匹配，一个偏语义匹配，一个偏具体动作）"。

## 观察 7：改写 prompt 的"动作词必须包含"规则只对单个 query 起作用，跨 query 不强制

- **位置**：`agents.py:383`
  ```
  "如果问题包含具体操作/动作（如'签派员放行检查'），query 必须包含这些动作词"
  ```
- **改写输出**：要求**每个** query 都含动作词。
- **检索接收**：5 个 query 全部含动作词 → 检索结果**全部围绕动作词召回** → 丢失"主体识别"和"法规定位"角度的召回。
- **差距**：动作词保证不是错的，但 5 个 query 都含动作词 = 没多样性。理想是 1-2 个 query 偏动作词，1-2 个 query 偏法规名（让 keyword routing 触发），1-2 个 query 偏主体角色。
- **量化影响**：影响难量化，但从 `KEYWORD_ROUTING_ENABLED=True`（`config.py:84`，消融v3: +18pp）看，**法规定位**比动作词重要得多。改写 prompt 的"动作词规则"可能拖累了法规定位路径。
- **建议**：把规则改成"至少 1 个 query 包含动作词"而不是"每个 query 都必须包含"。

## 综合判断

**改写对召回率的真实贡献**：

1. **改写 prompt 是召回率提升的主因**（估算贡献 50-60%）：5 个 query 并行检索时，召回面比单 query 大 3-5 倍。**这是 bot 答 Q19 找 article 31 的真正原因**。
2. **`KEYWORD_ROUTING` + `TREE_GENERIC_ARTICLE_PENALTY` + `law_hints` 是次要贡献**（估算 30-40%）：法规路由把"拒载"映射到"公共航空运输旅客服务管理规定"，比 query 改写还重要。
3. **DecompositionAgent 几乎不贡献**（< 5%）：触发条件几乎不满足。
4. **`QUERY_GATE` 贡献的是鲁棒性，不是召回率**（约 5%）：让改写 LLM 失败时仍能跑。

**对检索的对齐问题**：
- **改写机制 vs 检索系统 70% 对齐**。检索的 5 个路径（Phase 0 直读、BM25、Vector、KeywordRouting、Tree）只用到 2 个（Tree + KeywordRouting）。Phase 0 直读因为 prompt 没强制 article_hints 格式浪费，BM25+Vector 路径走的是 query → 树路径的间接通道。
- **gold vs bot 70% 对齐**。`test_gold_coverage.py:133-137` 用空主体/争点触发改写，系统性低估 bot 能力。
- **改写 prompt 的 prompt 工程是最大瓶颈**。5 个观察（1/2/3/6/7）全部指向 prompt 设计：article_hints 格式、sub_questions 强制、query 数量上限、query 多样性、动作词规则。

**最大单点优化**（如果只能改一处）：改写 system prompt 加 2 行——
1. `article_hints 必须形如"完整法规名第X条"，例：["公共航空运输旅客服务管理规定第三十一条"]`
2. `如果问题涉及 ≥2 个法规领域，sub_questions 必须拆分为 2-4 个子问题`

这两行直接修好 Phase 0 直读（贡献 10-15pp）和 DecompositionAgent（贡献 5-10pp），合计 15-25pp 的忠实率提升。

**Q19 漏 article 31 的根因**：
- 单 query "什么情况下航空公司可以拒载" → 关键词"航空公司""拒载"被 BM25 命中到"航空货物运输"的运输总条件（article 6-25，**因为这些条款频繁出现"运输"+"承运人"+"旅客"**），挤掉了 article 31 的位置
- 改写后 multi-query 之一命中"旅客服务 + 拒载 + 第三十一条"组合，把 article 31 顶到 top-10
- gold 评测用空 subject/issue 改写，**改写质量比 bot 差**，反而误判 bot 答错（虽然 bot 实际答对了）
