# Tree Retrieval 召回率低 — 根因诊断

**报告日期**：2026-06-03
**复现方式**：直接调用 `search_index_tree(...)`、`TreeRetriever.search(...)`、`_score_hints(...)`，并对照 `test30_20260603_190237.json` 的实际输出。
**结论摘要**：LeMAJ.supported 42% 低，**40% 来自 Level 2 找不到具体条款**，**~60% 来自 hint/路由把无关法规条款拉到顶部**。Level 0 在 hint 介入后通常选对法规，**真正坏在 Level 2 的索引语料质量 + hint 加分计算**。

---

## 根因 1：TreeRetriever 索引文本只用 title+summary，丢弃了 node.text

- **位置**：`legalbot/tree_retrieval.py:25-28`、`legalbot/parser.py:63-67`、`legalbot/parser.py:129`
- **机制**：
  - `_node_text(node) = grandparent_title + parent_title + node.title + node.summary`，**完全不包含 `node.text` 完整正文**。
  - `summary` 由 `_summary_from_text(text)` 生成（`parser.py:63-67`），逻辑是 `split_sentences(text)[0][:120]`。
  - `split_sentences` 用 `re.split(r"[。！？；\n]+", text)`（`utils.py:57`）—— **`\n` 与句号同等对待**，PDF 转换出来的换行符把句子切成碎片。
  - 实测：article:40（"旅客的托运行李延误到达的,承运人应当及时通知旅客领取..."）的 summary 是 `"第四十条 旅客的托运行李延误到达的,承运人应当及"`（24 字），**关键词"通知"、"领取"、"丢失"、"赔偿"全部丢失**。
  - 全语料统计：**10402 个 article 中 4473 个（43.0%）summary < 30 字符但 text > 80 字符**。
- **症状实例**：
  - **Q20** 行李丢失：`公共航空运输旅客服务管理规定` 第40条/第41条（行李延误/丢失赔偿）的 vector 编码字符串里没有"丢失"、"赔偿"，无法在全语料 BM25/Vector 中跑赢"民用航空法第15条 国籍标志"等噪声条款。
  - **Q06** EDTO：121.715/121.714/121.718（EDTO 系列）召回正常，但其他法规如"涡轮发动机燃油排泄规定"也因 token "EDTO" 不存在、靠路由意外命中。
- **影响范围**：**43% 条款语义信号残缺**——影响所有"用户问题用动词/形容词描述、答案条款用名词性 title 表达"的题目。这是召回率低的**第一性根因**。
- **修复建议**：
  - 短期：把 `_node_text` 改成包含完整 `node.text`，给 vector 模型截断到 512 tokens；BM25 不截断。
  - 中期：重写 `_summary_from_text`，先把多行连接成单行再 split sentences（先 `text.replace("\n", "")` 再 split），保证 summary ≥ 句号边界。
  - 同时检查 `hybrid_retrieval.VectorIndex.build:78`——它也漏了 `node.text`，要一并修。

## 根因 2：`_score_hints` 法规级广播给所有 article 相同 hint 分

- **位置**：`legalbot/retrieval.py:354-370`
- **机制**：
  ```python
  if law_hints and any(_law_matches(law, hint) for hint in law_hints):
      score += 2.0
  ```
  只要 article 所在法规命中任意 law_hint，**该法规所有 article 都得到 +2.0 hint_score**。`article_hint` 命中才额外 +3.5，但 LLM 经常不给 article_hints。
- 然后在 `rrf_fuse`（`hybrid_retrieval.py:158-160`）：
  ```python
  scores[key] += w_hint * min(hint_s / 12.0, 1.0)  # 0.30 × (2.0/12.0) = 0.0500
  ```
  - hint 加分 = **0.0500**
  - BM25 rank=1 加分 = `0.35/(15+1) = 0.0219`
  - Vector rank=1 加分 = `0.35/(15+1) = 0.0219`
  - **hint 加分 > BM25 排第 1 加分的 2.3 倍，> BM25+Vector 都排第 1 的总和**
- **症状实例**：
  - **Q20** 双 hint（`公共航空运输旅客服务管理规定` + `民用航空法`）场景下实测：
    ```
    [0.0557] 公共航空运输旅客服务管理规定 | article:34（行李条款，正确）
    ...
    [0.0500] 中华人民共和国民用航空法 | article:1（纯 hint，BM25/vec 都 0）
    [0.0500] 中华人民共和国民用航空法 | article:2（同上）
    [0.0500] 中华人民共和国民用航空法 | article:3（同上）
    ```
    实际生产输出的 16 篇 evidence，**6+ 篇是民用航空法第 6/7/8/9/107/22 条，全部靠 0.0500 纯 hint 拉到 top**。
  - **Q13** 无人机：LLM 推荐 `民用航空法`，导致民用航空法所有条款获得 +0.05，"机场净空"等无关条款排到前面。
- **影响范围**：**所有 LLM 给出 ≥2 个 law_hint 的题目**——根据 30 题 plan 推断约占 80%。
- **修复建议**：
  - **删除 line 366-369 的法规级 +2.0**，改为只在 article_hints 命中条款时加分。
  - 或保留法规级加分但把 `min(hint_s / 12.0, 1.0)` 改成 `min(hint_s / 60.0, 0.3)`，让 hint 不能超过 BM25+Vector 同名次的加分。
  - 或把 hint 通道从 `rrf_fuse` 移到 Level 0（只用于法规筛选），Level 2 完全不掺 hint。

## 根因 3：`art_orphan_mask` 索引空间混淆（law_idx vs ch_idx）

- **位置**：`legalbot/tree_retrieval.py:283-289`、`legalbot/tree_retrieval.py:87-93`
- **机制**：
  - `_article_to_chapter` 数组对**有章节的法规**存 ch_idx（0~1165），对**扁平法规**存 law_idx（0~127）。两套索引混在同一个数组里。
  - Level 2 想把"扁平法规的孤儿文章"也纳入候选：
    ```python
    top_law_indices = [int(idx) for idx, _ in law_set[:config.TREE_ORPHAN_TOP_LAWS]]  # 5 个 law_idx
    art_orphan_mask = np.isin(self._article_to_chapter, top_law_indices)
    ```
  - 但 `top_law_indices` 里的值（例如 `[15, 20, 6, 35, 97]`）会**同时匹配上 ch_idx=15/20/6/35/97**——这些是**其他法规章节的全局序号**，毫不相关。
- **症状实例**：
  - 实测 Q20 场景，`top_law_indices = [21, 101, 104, 24, 106]`，`art_orphan_mask.sum() = 24`，其中**21 篇是 chapter-collision 误中**（仅 4 篇是真正的孤儿文章），错误率 87.5%。
  - 例如 ch_idx=6 = `中国民用航空应急管理规定 chapter:8`，其中的应急条款被错误纳入 Q20 的 Level 2 候选。
- **影响范围**：所有走 tree 检索的 query（默认模式）。**每次至少注入 10~20 篇无关条款** 进入 Level 2 BM25/Vector 排序。
- **修复建议**：
  - 重构 `_collect_tree`：给"孤儿文章"用单独的 `article_to_law` 字段（保持 law_idx 空间），与 `article_to_chapter`（ch_idx 空间）分开。
  - 或者在 `_add_article` 中标记 `is_orphan=True/False`，Level 2 用 `is_orphan & np.isin(law_idx, top_law_indices)` 双重条件过滤。
  - 最简方案：把扁平法规的孤儿文章 ch_idx 设成"为该 law 创建一个虚拟章节"，让所有 article 都挂在 chapter 上，彻底消除混合索引。

## 根因 4：关键词路由的 jieba 分词与 routes.json 不一致

- **位置**：`legalbot/retrieval.py:294-302`（`_tokenize_for_routing`）+ `legalbot/retrieval.py:313-336`（匹配逻辑）
- **机制**：
  - `_tokenize_for_routing` 用 jieba 切词，门槛 `len(w) >= 2`。
  - "无人机" 是 jieba 字典里的整词 → tokens = `{"无人机"}`。
  - routes.json 中 `民用无人驾驶航空器运行安全管理规则` 的 keyword `"民用无人驾驶航空器"` → tokens = `{"民用", "无人驾驶", "航空器"}`。
  - **二者 token overlap = 0**！匹配阈值 `max(2, len(kw_tokens)//2) = 2`，永远命中不了。
  - 同时，**其他路由的"经营许可申请"**（tokens=`{"经营", "许可", "申请"}`）反而和 "申请空域许可" 产生 overlap=`{"申请","许可"}` = 2，**误命中"中国民用航空国内航线经营许可规定"等无关法规**。
- **症状实例**：
  - **Q13** 无人机 query 路由命中的法规：
    ```
    中国民用航空国内航线经营许可规定  ← 经营许可+申请误中
    外国航空运输企业不定期飞行       ← 申请+许可
    外国航空运输企业航线经营许可规定  ← 同上
    运输机场使用许可规定             ← 许可
    民用航空空中交通管理规则         ← 空域
    ...
    ```
    "民用无人驾驶航空器运行安全管理规则" 反而只靠 fallback hardcoded `{"无人机"}` 才补救。
- **影响范围**：**所有简称-全称不一致的领域**——无人机/无人驾驶航空器、安检/安全检查、CCAR-XXX/规则名、航司/航空公司等。jieba 切分粒度不可控的口语词全部受影响。
- **修复建议**：
  - 把 routes.json 的 keyword 改成 **字符级子串包含** 而非 token overlap：`if any(kw in q_normalized for kw in route_keywords)`。
  - 或者把所有路由 keyword 跑一遍同义词扩展（"无人驾驶航空器" → 也加 "无人机"），写入 routes.json。
  - 或者用 SentenceTransformer 给路由做语义匹配（与法规向量比相似度 > 阈值），完全甩开 jieba 分词。

## 根因 5：关键词路由产生过多噪声 hints，hint 加分被稀释

- **位置**：`legalbot/retrieval.py:305-351`、`legalbot/law_routes.json`
- **机制**：
  - `_resolve_law_by_keywords` 把 routes.json 中所有匹配的法规**全部加进 law_hints**，没有 top-k 截断或相关度排序。
  - routes.json 中很多法规的 keywords 含通用词（"经营许可"、"申请"、"许可"、"管理"等），导致一个 query 经常匹配 5~10 个法规。
  - 即使有正确法规被命中，它的 hint 加分（+0.30 Level 0、+0.05 Level 2）与 4~9 个噪声法规获得相同加分。
- **症状实例**：
  - **Q02** 燃油路由命中 5 个法规：`国内投资民航业...名单`、`正常类飞机适航规定`、`民用航空统计管理规定`、`运输类飞机适航标准`、`大型飞机公共航空运输承运人运行合格审定规则`。
  - 5 个法规同时被 boost，121.657（CCAR-121 内 vector rank=1, BM25 rank=2）反而被"正常类飞机适航规定 23.2430 燃油系统"压过（适航规定文档体量小，内部 vector 信号绝对值高）。
- **影响范围**：所有问题域不明确的 query。统计 30 题 query 路由命中数：平均 ~4 个 hints，中位数 5。
- **修复建议**：
  - 给路由结果加 top-k 截断（保留 token overlap 分数最高的 2~3 个）。
  - 给 routes.json 中的 keyword group 加权重，"经营许可"等通用词权重 < 0.5。
  - 在 fallback 硬编码规则中给"主路由"（如"行李 + 赔偿"→`公共航空运输旅客服务管理规定`）标记 `primary=True`，命中主路由时**清空其他 hints**，避免噪声稀释。

## 根因 6：Phase 0 article_hints 直读绕过检索

- **位置**：`legalbot/retrieval.py:151-218`（`_direct_read_articles`）+ `legalbot/retrieval.py:401-405`
- **机制**：
  - `_direct_read_articles` 把 LLM 给的 `article_hints`（"第121.20条" 这种）**直接读出对应条款**，绕过 BM25/Vector，给 `score=1.0`（最高分）。
  - 然后这些直读结果**优先注入 `direct_evidence`，并在合并时排在前面**（`retrieval.py:428-438`）。
  - **若 LLM 推荐错了**条号，错误证据直接霸占 top_k 名额，反而把检索召回的正确条款挤出去。
- **症状实例**：
  - **Q06** EDTO：LLM 推荐 `article_hints = ["第121.20条", "第121.380条", "第121.761条"...]`（运行合格证、维修记录、罚则，与 EDTO 无关），Phase 0 直读 → 这些条款全部以 score=1.0 进入 top_k。
  - 实测纯 query 检索能正确返回 121.715/121.714/121.718（EDTO 系列），但被 article_hints 直读注入的垃圾挤掉。
- **影响范围**：所有 LLM 推荐 article_hints 的题目（约占测试中 60%）。
- **修复建议**：
  - 给直读结果加**事后验证**：用 cross-encoder 比对 query 和直读条款的语义相似度，< 阈值的直接 drop。
  - 或者直读结果不给 score=1.0，给个温和的分数（如 0.6），让它和检索结果在同一分布下排序。
  - 或者让 LLM 在 RewritePlan 中标记 article_hints 的"置信度"，仅高置信度才走直读。

## 综合判断

**Level 0 / Level 2 哪个更严重？**

| 层 | 主要问题 | 严重程度 | 影响题数估算 |
|----|---------|---------|------------|
| **Level 0**（法规选错） | 路由噪声 hints 稀释正确法规（根因 4、5） | 中 | ~30%（Q02, Q13, Q08 类） |
| **Level 1**（章节选错） | 索引文本残缺导致 BM25/Vector 失准（根因 1） | 中 | ~20% |
| **Level 2**（条款选错） | hint 法规级广播 + 孤儿索引混淆 + article_hints 误直读（根因 2、3、6） | **高** | ~50%（Q06, Q20, Q24 类） |

**召回率低的本质是 Level 2 找不到具体条款（占主因）**，而不是 Level 0 选错法规。其中：
- 当 Level 0 因 hint 选对法规时（如 Q20），Level 2 反而被 hint 加分（根因 2）和 orphan_mask 误注入（根因 3）压制，把同法规内的"总则/通用"条款拉到顶部，淹没了"行李运输"等具体章节的语义信号。
- 当 Level 0 没有 hint 介入时（关键词路由失败，如 Q13 早期），即使 BM25/Vector 单独能找到正确条款（根因 1 影响有限的题），LLM 的错误 article_hints（根因 6）仍能直接污染 top_k。

**修复优先级（信噪比从高到低）**：
1. **根因 2**（hint 法规级广播）—— 一次代码改动，预期可让 Q13/Q20 这类的正确条款重回 top，影响面最大。
2. **根因 6**（article_hints 直读不校验）—— 加一道 cross-encoder gate 即可，立竿见影。
3. **根因 3**（orphan_mask 索引混淆）—— 数据结构改动，但逻辑简单，bug 性质明确。
4. **根因 1**（索引文本残缺）—— 影响最深远，但需要重建索引缓存，工程量最大。
5. **根因 4 + 5**（路由 jieba 不一致 + 噪声 hints）—— 改 routes.json 匹配逻辑 + 加 top-k 截断。

**预期收益**：根因 2 + 6 修完，预估 Faithfulness.supported 从 42% 涨到 55~60%；再补根因 1 后到 65~70%；全部修完后到 75% 左右（剩余 25% 是 LLM 合成阶段的问题，不在检索层）。
