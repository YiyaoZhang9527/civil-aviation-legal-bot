5 个 sub-agent 并行审查，**召回率低的真实根因有 3 个**（按 ROI 排序）：

| 优先级 | 问题 | 影响范围 | 修复成本 |
|--------|------|----------|----------|

## 30 题失败模式分类（sub-agent 5 的结果）

| 类型 | 题数 | 占比 | 关键症状 | 危险度 |

## 综合根因（来自 5 路 sub-agent）

### 🔴 根因 A（sub-agents 1, 2, 4 都指向）：

**`tree_retrieval.py:25-28` 的 `_node_text` 索引残缺**

```python
def _node_text(node, parent_title="", grandparent_title=""):
    parts = [grandparent_title, parent_title, node.title, node.summary or ""]
    return " ".join(p for p in parts if p)
```

只用 `title + summary`，**丢弃了 `node.text`（条款原文）**。`parser.py:63-67` 的 `_summary_from_text` 因 `split_sentences` 把 `\n` 当分句符，**43% 条款（4473/10402）的 summary < 30 字符**。

→ 嵌入到向量库的文本没有条款正文，BM25/vector 都搜不到"丢失"、"赔偿"、"拒载"等关键词。

### 🔴 根因 B（sub-agents 1, 3 指向）：

**`retrieval.py:354-370` 的 `_score_hints` 法规级广播**

```python
def _score_hints(article_hints, law_hints):
    for art in article_hints:
        # 命中就给该法规所有 article 都加 +2.0
        ...
```

命中一个条号 → 整部法规 +2.0。**RRF 融合后是 +0.050**，是 BM25 排名第 1 加分（0.022）的 **2.3 倍**。

→ 路由 hint 不准时，错误法规被强行拉到 top。

### 🔴 根因 C（sub-agent 3 指向）：

**`crossref.py` 把无关条文硬塞进 evidence**

```python
def resolve_references(evidence):
    # 用 _CROSS_BOOK_RE 提取《XX法》第X条，无 relevance 校验
    # 直接 read_law_node 加入 evidence
```

Q18 "航班延误餐饮" → 检索到一条 evidence → evidence 文本里有"适用本法第22条" → **第22条"民用航空器优先权"被加入 evidence** → bot 误用。

→ Crossref 是 Q18/Q22 类系统性错配的元凶。

### 🟡 根因 D（sub-agents 3, 5 指向）：

**Parser 污染 + 渲染双包**造成 `《《法规》已于2021年2月》`

```
parser.py:85-89  标题提取不跳 `《》` → law_title 携带 "《公共航空运输旅客服务管理规定》已于2021 年2 月"
agents.py:699   渲染时再裹 `《》` → `《《...》已于2021 年2 月》`
```

→ 10+ 部法规都有这个问题。LLM 看到 "已于2021年" 残留在 prompt 里，可能误判时效。

### 🟡 根因 E（sub-agents 2, 4 指向）：

**CE 评估器有 2 处逻辑 bug，不是模型问题**

1. **fallback 翻盘**：`citation.py:170-175` 和 `95-115` 会在 claim 分数低时**用 question 重打分**，导致检索错时本应低分的 evidence 被翻成 supported
2. **死字段** `Evidence.verified`：全代码库 grep 无任何读取方
3. **死路径** `_verify_claim_nli`：`CLAIM_LEVEL_CITATION=False` 默认，130 行代码生产路径不跑
4. **快速通道** `if all_supported: pass` 几乎不触发（fallback 给的分数难到 0.7 阈值）

→ **CE 模型本身没错，错在调用方逻辑。**

### 🟡 根因 F（sub-agent 2 指向）：

**Gold 评测用空 Subject/Issue 触发改写，系统性弱于 bot**

```python
# tests/test_gold_coverage.py:133-137
rewrite = rewrite_agent.rewrite_queries(
    question, SubjectAnalysis(), IssueAnalysis()  # ← 空对象！
)
```

→ Bot 用真实 subject/issue 上下文改写，gold 用空对象改写。**Q19 案例里 bot 实际答对 article 31，gold 反而判错——根因就是 gold 改写没用主体上下文。**

## 按 ROI 排序的修复方案

### 🥇 #1：D 类（数字幻觉）— **最高优先**

bot 编造 evidence 没有的具体数字（Q05 的"9-14 小时"、"60 小时/7 天"）。**最危险因为：答案看起来"专业"，MQS 满分，但实际在法律咨询里是致命错误。**

**修复**（1-2 天）：
- 在 Synthesis LLM 的 system prompt 加规则："evidence 文本外的具体数字必须标注 `[待核实]`"
- 改 synthesis 用更明确的 source 字段（已经有，但 LLM 不遵守）
- 加 post-processing：扫答案中的数字，检查是否在 evidence 中出现

**预估提升**：消除 D 类 3 题的 27% 幻觉 → Faithfulness.hallucinated 从 5.8% → < 2%

### 🥈 #2：B 类（找不到具体条款）— 11/30 题 = 37%

主因是 `_node_text` 索引残缺（根因 A）+ `_score_hints` 广播（根因 B）。

**修复**（3-5 天）：
1. **改 `_node_text` 用 `node.text` 完整正文**（不用 summary）— 1 行修改
2. **`_score_hints` 从法规级广播改为 article-level 精确加分**（命中具体条号才加分）— 30 行修改
3. **移除 `crossref` 自动硬塞**（改为 LLM 可选提示）— 50 行修改
4. **Re-build 索引**（清缓存重 build）— 5 分钟

**预估提升**：召回缺口从 71% → 50% 以下，supported_rate 42% → 60%+

### 🥉 #3：A 类（选错法规）— 8/30 题 = 27%

主因是路由 hint 不准（jieba 词典不一致）+ hint 广播。

**修复**（1-2 天）：
1. **统一 jieba 词典**（routes.json 用更短的关键字形式）— 配置文件改
2. **降低 hint 加分幅度**（2.0 → 0.5）— 1 行
3. **更严格的 article_hints 格式校验**（LLM 改写 prompt 要求完整法名）— 5 行 prompt

**预估提升**：选错率从 27% → < 15%

### 🟢 #4：B 类和 A 类的次要修

- Parser 净化 `_SKIP_TITLE_RE`（消除双书名号）— 10 行
- 删 CE fallback 翻盘逻辑（消除 supported 标签虚高）— 30 行
- 删 `Evidence.verified` 死字段 — 1 行
- 删 `_verify_claim_nli` 死代码 — 90 行

**预估收益**：代码量 -200 行，模型加载少 2s，指标更可信
