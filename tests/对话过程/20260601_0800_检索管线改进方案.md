# 检索管线改进方案 — 同时提升一致性和准确性

**时间:** 2026-06-01
**基于:** 代码审查 + RealRoute/Adaptive Query Routing 论文

## 一、当前管线全貌

```
用户 query
  │
  ▼
[LLM] SubjectAgent → [LLM] IssueAgent → ClarificationAgent(规则)
  │
  ▼
[LLM] RewriteAgent → A1查询门控(原始query保底) → [LLM] DecompositionAgent
  │
  ▼
RetrievalAgent.retrieve()
  │
  ├─ A2 关键词路由(_resolve_law_by_keywords) → law_hints  ← 【问题源头】
  ├─ Phase 0: 直读法条(正则匹配)
  └─ Phase 1: Tree检索(tree_retrieval.py)
       │
       ├─ Level 0 法级: Dense(gte-large-zh) + BM25(jieba) + RRF → 30个候选法
       │    └─ A2 hint boost: 匹配到的法规+0.3分(已改为软加权)
       │
       ├─ Level 1 章级: Dense + BM25 + RRF → 15个候选章(限在候选法内)
       │
       ├─ Level 2 条级: Dense + BM25 + Hint + RRF → top_k个候选条
       │
       └─ Cross-Encoder精排(bge-reranker-v2-m3) → RERANKER_MIN_SCORE(0.1)过滤
  │
  ▼
回读原文 → 交叉引用补全 → [CE] CitationAgent → ConflictAgent → [LLM] SynthesisAgent
  │
  ▼
[LLM] ReflexionAgent → [质量不足] 补搜重试(最多2轮)
```

**LLM 调用次数:** 最少6次，最多约10次
**本地模型:** gte-large-zh(向量) + bge-reranker-v2-m3(精排) + BM25(jieba)
**SPLADE:** 不存在（CLAUDE.md 记录有误）
**LLM:** DeepSeek v4-flash

## 二、当前问题诊断

### 问题1: A2 关键词路由质量差（已止血）
- law_routes.json 缺少用户自然语言关键词
- 匹配阈值过于宽松（2个token重叠就命中）
- 已改为软加权（+0.3），但路由表本身质量仍然差

### 问题2: Level 0 法级检索精度不足
- gte-large-zh 对法律短文本区分度不够
  - "备降" 和 "燃油排泄" 语义相似度偏高
  - 法级节点文本只有标题+summary，信息量少
- BM25 只能做精确词匹配，无法扩展同义词
  - "值勤期" 查不到 "飞行时间限制"
  - "备降" 查不到 " alternate airport"
- 没有学习型稀疏检索（SPLADE），BM25 是唯一的稀疏通道

### 问题3: 检索失败无动态恢复
- Level 0 路由错误 → Level 1/2 全在错误法规内展开
- Cross-Encoder 精排过滤后为空 → 直接返回空列表，无 fallback
- ReflexionAgent 可以补搜但仍在相同框架内重试，无法换法规

### 问题4: RRF 参数不适合小语料
- RRF_K=60 是 TREC 规模（数千文档）的默认值
- 我们 Level 0 只有 129 个法规，K=60 过于平坦
- 论文建议小语料用 K=10-20

## 三、改进方案设计

### 设计原则

借鉴两篇论文的核心思想：

**RealRoute**: 不预测路由，而是并行检索+验证+动态调整
**Adaptive Query Routing**: 法律领域 Tree Reasoning 最优，但不同复杂度应走不同路径

同时结合我们现有架构的实际情况：
- Tree架构已被论文验证为法律领域最优，保留
- A2关键词路由是失败的尝试，废弃
- Cross-Encoder精排是正确的，但需要更好的输入

### 改进1: Retrieve-then-Verify 动态路由（借鉴 RealRoute）

**核心思想**: 不再在检索前预测"查哪个法规"，而是先广检索，再用 Cross-Encoder 验证质量，验证不通过就扩大范围重查。

```
当前流程:
  Level 0 (30个候选法) → Level 1 (15个候选章) → Level 2 (top_k候选条) → Cross-Encoder精排
  ↑ 如果Level 0就错了，后面全错

改进后:
  Level 0 广检索(全部129法) → Level 1 → Level 2 → Cross-Encoder精排
  │
  ├─ [精排结果>=3条] → 正常流程
  │
  └─ [精排结果<3条] → 触发扩展重检索:
       ├─ 分析精排分数分布，识别"检索可能偏了"
       ├─ Level 0 扩大候选(TREE_TOP_LAWS 30→60)
       ├─ 放宽 Level 1/2 剪枝阈值
       └─ 重新 Level 1 → Level 2 → Cross-Encoder精排
```

**实现位置**: `retrieval.py` `_tree_search()` 的精排后检查（行581-597之后）

**具体逻辑**:
```python
# 在 _tree_search() 的精排步骤后加入:
if len(reranked) < 3:
    # 扩展重检索：增大Level 0候选数，放宽阈值
    tree_results = tree.search(query, top_k=top_k * 2,
                               hint_law_ids=None,  # 不限法规
                               hint_scores=None,
                               tree_top_laws=60)    # 扩大到60法
    # 重新精排...
```

**收益**: 
- 解决"Level 0 路由错误导致全盘失败"的问题
- Cross-Encoder 作为验证器而非过滤器
- 借鉴 RealRoute: "证据级验证而非预路由"

### 改进2: Level 0 增加学习型稀疏检索

**核心思想**: 在 Level 0 的 RRF 融合中加入第三个稀疏通道，自动扩展同义词。

**方案选择**: 不引入新模型（SPLADE需要额外训练），而是用**现有的 Cross-Encoder 做法级粗筛**。

```
当前 Level 0:
  Dense向量(gte-large-zh) + BM25(jieba) → RRF → 30个候选法

改进后 Level 0:
  Dense向量(gte-large-zh) + BM25(jieba) + 法规摘要Cross-Encoder粗筛 → RRF → 30个候选法
```

**具体实现**:
1. 为每个法规预计算一条代表性文本（标题 + 章节概要 + 高频条文摘要，约200字）
2. 查询时，用 Cross-Encoder 对 query 与 129 条法规摘要打分
3. Cross-Encoder 分数作为第三路信号参与 RRF 融合

**为什么不用SPLADE**:
- 中文SPLADE模型生态不成熟，没有现成的高质量模型
- 需要额外训练和推理开销
- Cross-Encoder 我们已经有了，且精度更高

**收益**:
- Cross-Encoder 的语义理解远强于 BM25 的词匹配
- "备降" 和 "大型飞机公共航空运输承运人运行合格审定规则" 的摘要，Cross-Encoder 能正确判断相关性
- 无需维护任何路由表

**代价**:
- 每次查询多一次 Cross-Encoder 推理（129个 pair）
- 可以离线预计算法规向量，查询时只计算 query 侧
- 实测 bge-reranker-v2-m3 在 CPU 上推理 129 pair 约需 0.5-1秒

### 改进3: RRF 参数调优

```
当前: RRF_K=60 (TREC规模默认值)
改进: RRF_K=15 (129部法规的小语料适配)

当前: HYBRID_WEIGHT_BM25=0.35, HYBRID_WEIGHT_VECTOR=0.35, HYBRID_WEIGHT_HINT=0.30
改进: 取消 Hint 通道，改为 BM25=0.30, Vector=0.35, CrossEncoder=0.35
```

### 改进4: 废弃 A2 关键词路由

```
当前: _resolve_law_by_keywords() 基于 law_routes.json 路由
改进: 完全删除 A2 路由调用，由 Level 0 的 Dense+BM25+CrossEncoder 三路融合替代
```

A2 路由表是手工维护的规则，已被 Level 0 的语义检索完全覆盖。Cross-Encoder 粗筛天然泛化，不需要任何关键词。

### 改进5: ReflexionAgent 增加跨法规补搜

```
当前: ReflexionAgent 补搜时仍在原 RetrievalPlan 框架内重试
改进: 补搜时清除 law_hints 限制，在全部129法内重新检索
```

这就是 RealRoute 的 "reflection loop" — 每次重试更新 failure context，避免重复相同的无效路由。

## 四、改进后的完整管线

```
用户 query
  │
  ▼
[LLM] SubjectAgent → IssueAgent → ClarificationAgent
  │
  ▼
[LLM] RewriteAgent → A1查询门控 → DecompositionAgent
  │
  ▼
RetrievalAgent.retrieve()
  │
  ├─ Phase 0: 直读法条(保留)
  │
  └─ Phase 1: Tree检索(v2)
       │
       ├─ Level 0 法级: Dense + BM25 + CrossEncoder粗筛 → RRF(K=15) → 30个候选法
       │    └─ 无A2路由，纯语义检索
       │
       ├─ Level 1 章级: Dense + BM25 + RRF → 15个候选章
       │
       ├─ Level 2 条级: Dense + BM25 + RRF → top_k个候选条
       │
       └─ Cross-Encoder精排 → RERANKER_MIN_SCORE过滤
            │
            ├─ [>=3条通过] → 正常流程
            └─ [<3条通过] → 扩展重检索(扩大Level 0候选)
  │
  ▼
[CE] CitationAgent → ConflictAgent → [LLM] SynthesisAgent
  │
  ▼
[LLM] ReflexionAgent → [质量不足] 跨法规补搜重试
```

## 五、预期收益

| 指标 | 当前 | 改进后预期 | 原因 |
|------|------|-----------|------|
| 法级路由准确率 | 低(依赖A2关键词) | 高(CE语义理解) | Cross-Encoder远强于关键词匹配 |
| 跨法规泛化 | 差(路由表覆盖有限) | 好(纯语义) | 无需维护任何规则 |
| 精排后空结果率 | 高(路由错误→全错) | 低(Retrieve-then-Verify) | 扩展重检索兜底 |
| 交叉引用召回 | 未测 | 100%(论文验证) | Tree架构在法律领域已验证 |
| 单次查询延迟 | ~15s | ~16-17s | 增加0.5-1s CE粗筛 + 可能的扩展重检索 |

## 六、实施顺序

1. **第一步**: 废弃A2路由 + RRF_K调优 → 跑30题验证基线恢复
2. **第二步**: Level 0 增加CE粗筛 → 跑30题对比
3. **第三步**: Retrieve-then-Verify扩展重检索 → 跑30题对比
4. **第四步**: ReflexionAgent跨法规补搜 → 跑30题对比
5. **每步对比**: evidence_count、citation_count、supported_rate、答案质量
