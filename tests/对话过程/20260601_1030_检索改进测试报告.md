## 三、统计分析

### 排除澄清拦截后（6题正常检索）

| 指标 | 基线 | 当前 | 变化 |
|------|------|------|------|
| evidence>0 率 | 6/6 (100%) | 6/6 (100%) | 持平 |
| 平均 evidence 数 | 10.7 | 11.3 | +0.6 |
| 平均 supported 数 | 5.2 | 5.0 | -0.2 |
| 平均耗时 | 179s | 265s | +48% |

### 澄清拦截（3题）

Q01、Q02、Q08 被前置的 ClarificationAgent 拦截，返回 "缺少关键信息"。
- 基线中这3题正常回答（evidence=12, 12, 4）
- 当前版本在 9-10秒内返回拒绝回答
- **原因：与检索改进无关，是 ClarificationAgent 判断逻辑问题**

### Retrieve-then-Verify 触发情况

当前9题中未观察到扩展重检索被触发（所有正常检索题 evidence>=6）。
效果待更多测试验证。

## 四、新发现的问题

### 问题1: 进程在 Q12 时崩溃

两次运行都在处理约11题后进程消失（无 traceback、无 OOM）。
- 内存充足（21G 可用）
- 可能是 LLM API 超时 + 异常处理链的 bug
- 已在 answer() 中加入 traceback.print_exc()

### 问题2: ClarificationAgent 过度拦截

3/9 题被拦截。这些题在基线中正常回答。
- Q01 "备降" → 被要求说明 CCAR-121 还是 CCAR-135
- Q02 "燃油量" → 被要求说明航空公司还是通航
- Q08 "结冰条件" → 被要求说明运行规范中的具体规定

这是 ClarificationAgent 的判断阈值问题，不是检索问题。

### 问题3: 耗时增加 48%

平均耗时从 179s 增加到 265s。原因：
- Retrieve-then-Verify 逻辑增加了额外的精排判断
- 但核心耗时仍在 LLM 调用（6-10次/题）

## 五、结论

**检索管线改进有效但有限：**
- evidence 数量基本持平（10.7 → 11.3）
- supported 数量基本持平（5.2 → 5.0）
- 废弃 A2 路由后，检索不再被错误过滤，管线恢复到基线水平
- RRF_K 调优和 Retrieve-then-Verify 作为安全网存在

**核心瓶颈不在检索，在更上层：**
1. ClarificationAgent 过度拦截（3/9 题直接拒绝）
2. 进程崩溃（可能跟 LLM API 或内存管理有关）
3. LLM 调用耗时占 80%+（6-10 次 API 调用/题）

**CE 粗筛待 GPU 加速后验证：**
- CPU 上 129 pair 需 6+ 秒，不可接受
- GPU 上预计 <0.5 秒，值得开启
