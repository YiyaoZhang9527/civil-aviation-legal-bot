## 1. 完整性评测

### 1.1 LeMAJ 的 Legal Data Points (LDP) 方法 [核心推荐]

**来源**: [LeMAJ (Legal LLM-as-a-Judge)](https://arxiv.org/html/2510.07243v1) — Robin AI + Amazon Web Services, 2024

这是目前对"完整性"最精确的可操作方案。

**核心方法**:
- 将答案分解为独立的 **Legal Data Points (LDPs)** —— 自包含的信息单元
- 每个LDP用四级标签标注:
  - `<correct>`: 事实正确且相关
  - `<incorrect>`: 事实错误或幻觉
  - `<irrelevant>`: 事实正确但不相关 (相当于 false positive)
  - `<missing>`: 应该包含但遗漏的信息 (相当于 false negative)

**推导指标**:
```
Correctness = #Correct / (#Correct + #Incorrect)     -- 事实准确率
Precision   = #Correct / (#Correct + #Irrelevant)    -- 相关率
Recall      = #Correct / (#Correct + #Missing)       -- 完整率!
F1          = 2 * Precision * Recall / (P + R)       -- 综合分数
```

**关键洞察**: Recall 直接衡量完整性 —— 缺失的关键信息被显式捕获。F1平衡了"说了该说的"和"没漏该说的"。

**与我们系统的对接**:
- 我们已有 claim-level faithfulness 分解，可复用同一分解粒度
- 只需增加 `<missing>` 标签的检测：让LLM基于问题+知识库判断答案遗漏了什么
- 可直接给出数值化的完整性分数

### 1.2 RAGAS Context Recall

**来源**: [RAGAS 官方文档](https://docs.ragas.io/en/stable/concepts/metrics/available_metrics/context_recall/)

**公式**:
```
Context Recall = 参考答案中被检索上下文支持的claims数 / 参考答案中总claims数
```

**局限**: 需要参考答案作为proxy。Legal RAG Bench的作者Isaacus批评了这一做法，认为法律领域的参考答案成本极高。

### 1.3 BCG 语义测试覆盖率 (Semantic Test Coverage)

**来源**: [BCG X - Testing the Tests](https://www.bcg.com/x/the-multiplier/how-measure-rag-evaluation-completeness)

**方法**:
1. **映射知识库**: 将知识库chunk化，聚类为语义主题
2. **映射测试题**: 将测试题投影到同一语义空间
3. **过滤无效题**: 用异常检测移除偏题
4. **计算覆盖率**: 三种分数 (basic / content-weighted / multi-topic)
5. **定位盲区**: 找出覆盖率低的cluster，生成新测试题
6. **持续监控**: 随知识库更新重复检测

**适用场景**: 这解决的不是"单个答案是否完整"，而是"测试集是否覆盖了所有知识领域"。适合我们扩充测试集时使用。

### 1.4 Legal RAG Bench 的全因子实验设计

**来源**: [Isaacus Legal RAG Bench](https://isaacus.com/blog/legal-rag-bench)

**方法**: 三元组 (Correctness, Groundedness, Retrieval) 的全因子设计:
- **Correctness**: 回答是否蕴含正确答案
- **Groundedness**: 回答是否基于检索到的文档 (不管文档是否相关)
- **Retrieval accuracy**: 是否检索到了相关段落

**错误归因分类**:
| 错误类型 | 条件 | 含义 |
|---------|------|------|
| 幻觉 | Groundedness=0 | 模型编造了上下文中没有的事实 |
| 检索错误 | G=1, C=0, R=0 | 上下文不够，模型基于不完整信息生成了错误答案 |
| 推理错误 | G=1, C=0, R=1 | 正确信息被检索到但模型推理出错了 |

**关键发现**: "检索性能决定了法律RAG系统的性能上限。强检索可以弥补弱推理，但强推理无法弥补弱检索。"

## 2. 场景化测试设计

### 2.1 Harvey LAB (Legal Agent Benchmark) [标杆]

**来源**: [Harvey LAB](https://www.harvey.ai/blog/introducing-harveys-legal-agent-benchmark)

**规模**: 1,200+ agent tasks, 24 practice areas, 75,000+ expert rubrics

**设计原则**:
1. **Client matter-centric**: 模拟真实律所工作分配
2. **短指令 + 复杂环境**: 指令平均仅50词，但环境包含混合的关键/外围文件
3. **开放式任务**: 不是简单的Q&A，而是"分析变更控制条款，评估风险，准备备忘录"
4. **All-pass grading**: 任务仅在所有标准通过时才算完成 —— "发现10个风险中8个的报告不是80%有用，而是实质上不完整"
5. **原子化评估标准**: 每个rubric是二元的pass/fail，覆盖事实、结论、引用、严重性评级、建议、截止日期、金额、格式

**M&A任务示例**:
- 输入: 50词指令 + 虚拟数据室(8份核心合同+周边材料)
- 要求: 识别变更控制条款 → 评估风险 → 推荐下一步 → 出具备忘录
- 评估: 57条标准，覆盖9个法律问题，每个问题4-9条标准

**对我们的启示**: 民航法律问答可以按此思路设计:
- 不是问"XX法第X条怎么说"，而是给一个航空事故场景，要求综合多部法规分析责任
- 评估标准应该是原子化的、二元的

### 2.2 Vals AI VLAIR (Vals Legal AI Report)

**来源**: [VLAIR Feb 2025](https://www.vals.ai/industry-reports/vlair-2-27-25), [VLAIR Oct 2025](https://www.vals.ai/industry-reports/vlair-10-14-25)

**方法**:
- 4个法律AI工具 vs 律师对照组
- 7种法律任务: Data Extraction, Document Q&A, Document Summarization, Redlining, Transcript Analysis, Chronology Generation, EDGAR Research
- **三个加权评分标准** (具体权重未公开)
- 200个美国法律研究问题

**关键设计特点**:
- 对照组使用真实律师，而非参考答案
- 测量的是AI相对于人类律师的表现，而非绝对准确率
- 2025年10月报告发现: AI在法律研究准确性上已超越律师

### 2.3 Stanford RegLab Legal RAG Benchmarks

**来源**: [RegLab Legal RAG Benchmarks](https://reglab.github.io/legal-rag-benchmarks/)

**两个基准**:
- **Bar Exam QA**: 基于真实律师考试题
- **Housing Statute QA**: 基于真实住房法规查询场景

**设计方法**: 从真实法律研究任务出发，而非从模型能力出发。

### 2.4 民航法律场景化测试设计建议

基于以上方法论，民航法律问答系统的测试集应包含:

| 场景类型 | 描述 | 示例 |
|---------|------|------|
| 单法条查询 | 事实型问题 | "民用航空器适航管理的审批机关是谁？" |
| 跨法条综合 | 需综合多部法规 | "发生航空器事故后，哪些部门有权参与调查？各自职责是什么？" |
| 情景应用 | 给定具体案例 | "一架B737在着陆时冲出跑道，请分析可能涉及的法律责任" |
| 比较分析 | 法规对比 | "《民用航空法》和《安全生产法》在事故调查方面有何异同？" |
| 流程指导 | 步骤型回答 | "申请航线经营许可的完整流程是什么？" |
| 不可回答 | 知识库外问题 | "美国FAA对无人机管理有哪些规定？" |
| 过时法规 | 已废止/修改 | 引用已失效的法规版本测试 |
| 矛盾法规 | 存在冲突的条款 | 测试系统如何处理法规间的矛盾 |

## 3. 参考答案/金标准构建

### 3.1 LeMAJ 的无参考方案 [推荐]

**核心思路**: 不需要预先创建参考答案，而是让LLM judge直接基于原文档评估:
1. 分解答案为LDPs
2. 对照源文档判断correct/incorrect
3. 对照问题判断relevant/irrelevant
4. 基于源文档识别missing信息

**优势**: 无需人类专家预先标注参考答案，但评估结果与人类专家相关性更高。

**验证数据**: 与人类金标准相关性 (Pearson):
- Relevance: 0.61 (vs DeepEval 0.079)
- Correctness: 0.56 (vs DeepEval 0.43)

### 3.2 LLM-as-Judge + 人类元评审

**来源**: 综合 [DeepEval](https://deepeval.com/guides/guides-llm-as-a-judge), [HuggingFace Cookbook](https://huggingface.co/learn/cookbook/llm_judge)

**推荐流程**:
1. 用强模型(如Claude Opus)生成参考答案
2. 法律专家审核/修正参考答案 (只修正错误，不重写)
3. 用LLM-as-Judge对系统输出进行自动化评分
4. 定期抽样人类评审，校准LLM judge的准确性
5. 使用 Krippendorff's alpha 测量标注者间一致性

**半自动化金标准创建**:
```
1. 从知识库中抽取关键法条和条款
2. 用LLM基于法条生成问答对
3. 人工审核问答质量 (3人标注，多数投票)
4. 将审核通过的问答对加入测试集
5. 定期重新评估 (法规更新时)
```

### 3.3 Harvey LAB 的原子化rubric方法

每个评估标准是:
- **二元的** (pass/fail)
- **原子的** (只检查一件事)
- **具体的** (精确到"是否识别出Pinnacle许可证在控制权变更后将独家转为非独家")

示例rubric结构:
```
风险1: Pinnacle许可证独家转非独家
  - 是否识别出该条款? [Y/N]
  - 是否正确评定严重性? [Y/N]
  - 是否给出财务影响估算? [Y/N]
  - 是否建议下一步行动? [Y/N]
风险2: ...
```

## 4. 端到端RAG评测框架

### 4.1 RAGAS 核心指标体系

**来源**: [RAGAS](https://docs.ragas.io/en/stable/concepts/metrics/available_metrics/)

| 指标 | 衡量什么 | 需要什么输入 | 是否需要参考答案 |
|------|---------|------------|---------------|
| Faithfulness | 答案是否基于检索上下文 | question + answer + contexts | 否 |
| Answer Relevancy | 答案是否与问题相关 | question + answer | 否 |
| Context Precision | 检索结果中相关文档排名是否靠前 | question + contexts | 是(参考上下文) |
| Context Recall | 是否检索到所有相关文档 | question + contexts + reference | 是 |
| Answer Correctness | 答案是否正确 | question + answer + reference | 是 |
| Answer Similarity | 答案与参考答案的语义相似度 | answer + reference | 是 |

**2025新增**: Agent metrics (task completion, tool correctness, step grading)

**关键实现细节**:
- Context Recall 将参考答案分解为claims，检查每个claim是否可归因于检索上下文
- Faithfulness 将答案分解为claims，检查每个claim是否可从上下文推导
- Answer Relevancy 通过生成该答案可能对应的问题来反向验证

### 4.2 分层评测: 检索 vs 生成

**来源**: Legal RAG Bench + RAGAS

**检索层评测**:
```
指标:
- Recall@K: 前K个结果中包含多少相关文档
- MRR: 第一个相关文档的排名倒数
- Context Precision: 检索结果中相关文档的比例
- nDCG: 考虑排序位置的检索质量

实现:
1. 为每个测试问题标注"应检索到的法条/条款列表"
2. 运行检索管线
3. 对比检索结果与标注列表
```

**生成层评测**:
```
指标:
- Faithfulness: 答案claims是否全部可从检索上下文推导
- Correctness: 答案是否事实正确
- Completeness (Recall from LeMAJ): 是否遗漏关键信息
- Relevancy: 答案内容是否都与问题相关

实现:
1. 固定检索结果 (使用金标准上下文)
2. 只评测生成模型的表现
3. 隔离检索vs生成的错误
```

### 4.3 Isaacus Legal RAG Bench 的全因子方法

**来源**: [Legal RAG Bench](https://isaacus.com/blog/legal-rag-bench) — 发布于2026年3月

**设计**:
- 4,876段落 + 100个手写复杂问题
- 全因子实验: embedding模型 x 生成模型 的所有组合
- 三元评估: (Correctness, Groundedness, Retrieval)

**关键方法论贡献**: 能够将错误归因到管线具体环节:
- 如果换embedding模型后correctness提升34分 → 检索是瓶颈
- 如果换生成模型后groundedness变化6分 → 生成是次要因素

## 5. 超越忠实度的答案质量

### 5.1 Stanford "Measuring What Matters" 质量评价体系

**来源**: [Stanford Legal Design Lab - JURIX 2024](https://justiceinnovation.law.stanford.edu/measuring-what-matters-a-quality-rubric-for-legal-ai-answers/)

**四个核心评价维度** (从46个用户访谈 + 21个法律专家评估得出):

| 维度 | 定义 | 优先级 |
|------|------|--------|
| **Accuracy** | 法律信息是否正确，高风险权利是否附带风险提示 | 必要但不充分 |
| **Actionability** | 是否提供用户可执行的具体步骤 | 比准确性更重要 |
| **Empowerment** | 是否帮助用户感到有能力处理问题 | 重要 |
| **Strategic Caution** | 是否避免不必要的恐惧或阻碍行动 | 重要 |

**关键发现**:
1. "准确性alone不够" —— 技术上正确但不可操作的答案价值低于"略不精确但高度可操作"的答案
2. "误导性答案比完全错误的更危险" —— 部分正确的答案让用户信任但可能导致关键步骤错误
3. "过度谨慎的免责声明有害" —— "请咨询律师"在大多数人无法获得律师时等于拒绝帮助
4. "引用不如可操作性重要" —— 用户更关心下一步做什么而非法律条文出处
5. "最好的AI回答像GPS" —— 诊断问题 → 指明路径 → 提供资源

### 5.2 Answerability (可回答性检测)

**来源**: [OpenReview - Fact-Level Answerability](https://openreview.net/forum?id=dNFDLAvk48), [Decoding AI](https://www.decodingai.com/p/rag-evaluation-6-metrics-framework)

**定义**: 系统是否正确判断了自己能不能回答某个问题。

**Fact-Level Answerability**: 一个多跳问题可回答，当且仅当推理链中的每一个必要事实（"跳"）在上下文中都存在。

**评测方法**:
1. 创建混合测试集: 包含可回答和不可回答的问题
2. 对每个问题标注: answerable (Y/N), 若N则缺失哪些事实
3. 测量系统:
   - 对可回答问题: 是否给出了正确答案
   - 对不可回答问题: 是否正确拒绝 (而非编造答案)
4. 计算 F1 score: 精确率(拒绝的都是该拒绝的) × 召回率(该拒绝的都拒绝了)

**对民航法律系统的具体实现**:
```
测试集构造:
- 30% 可回答问题 (知识库覆盖)
- 20% 部分可回答 (知识库只覆盖部分)
- 20% 不可回答 (知识库无覆盖)
- 15% 跨领域问题 (需要非民航法律知识)
- 15% 陷阱问题 (看起来相关但实际无关)

评测指标:
- Refusal Precision: 被拒绝的问题中，真正应该拒绝的比例
- Refusal Recall: 应该被拒绝的问题中，实际被拒绝的比例
- Answer F1: 可回答问题的正确率
```

### 5.3 实用性/可操作性评分

**来源**: Stanford Quality Rubric + 自定义设计

**评分标准**:
```
5分: 给出明确的行动步骤 + 引用具体法条 + 提供相关表格/流程链接
4分: 给出行动方向 + 引用法条
3分: 解释法律原则 + 建议咨询专业人员
2分: 只给出抽象法律原则
1分: 答非所问或误导
```

## 6. 对抗性/边缘案例测试

### 6.1 法律AI特有的对抗性测试类型

| 测试类型 | 描述 | 期望系统行为 |
|---------|------|-------------|
| **过时法规** | 引用已废止/修订的法规 | 明确指出已失效，引用现行版本 |
| **矛盾法规** | 同一问题在不同法规中有不同规定 | 指出矛盾，分析适用场景 |
| **法律推理陷阱** | 表面适用但实际不适用的法条 | 识别陷阱，给出正确分析 |
| **超出范围** | 知识库不覆盖的领域 | 拒绝回答或明确说明局限性 |
| **部分覆盖** | 知识库有相关信息但不完整 | 回答已知部分，说明信息不完整 |
| **模糊表述** | 模棱两可的问题 | 澄清问题或给出多种可能解释 |
| **复合问题** | 包含多个子问题 | 分别回答每个子问题 |
| **否定假设** | "XX不是YY对吗？" | 正确处理否定句式 |
| **数字敏感** | 涉及具体金额、日期、数量 | 精确回答，不能有偏差 |
| **时效性** | 询问法规生效时间 | 给出准确的时间线 |

### 6.2 Thomson Reuters CoCounsel团队的LLM评测方法

**来源**: [Thomson Reuters - Efficiently Evaluating LLMs for Legal Tasks](https://legal.thomsonreuters.com/blog/evaluating-llms-legal-tasks/)

**方法**:
1. 创建法律领域特定的评测任务
2. 对每个任务定义明确的评分标准
3. 包含"负例"测试 (系统应该拒绝或标记的输入)
4. 测量在不同法律任务类型上的一致性

### 6.3 民航法律对抗性测试集设计

**具体测试用例示例**:

```
1. 过时法规测试:
   Q: "根据《民用航空运输不定期飞行管理暂行规定》，申请不定期飞行需要哪些审批？"
   (该规定已被新法规替代)
   期望: 系统指出该规定已失效，引用现行有效法规

2. 矛盾法规测试:
   Q: "航空器事故调查中，谁负责组织调查？"
   (《民用航空法》vs《生产安全事故报告和调查处理条例》可能给出不同答案)
   期望: 系统指出两部法规的适用范围差异

3. 陷阱问题:
   Q: "私人无人机在自家院子上方100米飞行需要许可吗？"
   (看似简单，实际涉及空域分类、无人机重量、飞行高度多层规定)
   期望: 系统分情况讨论

4. 超范围问题:
   Q: "美国FAA Part 107对商业无人机有什么要求？"
   期望: 系统明确说明知识库不包含美国法规

5. 部分覆盖:
   Q: "国际航班上发生人身伤害，乘客如何索赔？"
   (涉及《蒙特利尔公约》+ 《民用航空法》，知识库可能只覆盖后者)
   期望: 回答已知部分，说明国际条约部分需另行查询
```

## 7. 中文法律AI评测体系

### 7.1 LexEval (NeurIPS 2024)

**来源**: [LexEval - NeurIPS 2024](https://neurips.cc/virtual/2024/poster/97832), [GitHub](https://github.com/cshaitao/lexeval)

**规模**: 23个任务, 14,150个问题 —— 目前最大的中文法律评测数据集

**核心设计**: **LexCog** 法律认知能力分类法
- 按认知复杂度分层
- 覆盖法律知识应用的多个维度
- 包含伦理考量评测

### 7.2 MSLR-Bench (IRAC框架)

**来源**: [MSLR-Bench](https://arxiv.org/html/2511.07979v3), [ModelScope](https://modelscope.cn/datasets/yuwenhan/MSLR-Bench)

**核心**: 首个中文多步法律推理基准，对齐 **IRAC框架**:
- **I**ssue: 识别法律争议点
- **R**ule: 识别适用法规
- **A**pplication: 将法规适用于事实
- **C**onclusion: 得出结论

**评测方法**: LLM评分 (Grade A/B/C)，由人类专家审核

**对我们的启示**: 可将IRAC框架应用于民航法律问答评测，检验系统是否完成了完整的法律推理链。

### 7.3 LAiW (COLING 2025)

**来源**: [LAiW GitHub](https://github.com/Dai-shen/LAiW)

**特点**: 基于法律三段论评测，三个能力层次
- 自定义指标 **SCULAiW**

### 7.4 CAIL 2025 (中国法律智能技术评测)

**来源**: [CAIL 2025](https://hub.baai.ac.cn/view/49378)

**6个评测任务**:
1. 司法考试客观题
2. 司法主观题考试
3. 智能体法庭模拟
4. 智能体模拟调解
5. 法律数值计算
6. 量刑情节识别与刑期预测

**趋势**: 从单一问答向智能体模拟演进

### 7.5 Super Legal Bench (中国司法大数据研究院)

**来源**: [司法大数据研究院](https://data.court.gov.cn/pages/modelEvaluation.html)

**规模**: 10万+法律测试集
**标准**: GB/T 25000.51, CNAS认证实验室
**评测维度**: 5大维度 (具体维度未公开)

### 7.6 上海交大智慧司法研究院评测体系

**来源**: [上海交大](https://zhsf.sjtu.edu.cn/info/1587/3994.htm)

**特点**: 面向真实司法任务的评测方法，多维度评估指标体系

### 7.7 中文法律AI评测的特殊挑战

| 挑战 | 描述 | 应对方案 |
|------|------|---------|
| 法规层级复杂 | 法律 > 行政法规 > 部门规章 > 规范性文件 | 明确标注法规层级和效力 |
| 地方性法规 | 各地有不同实施细则 | 测试系统对地域差异的处理 |
| 法规更新频繁 | 新法规出台、旧法规修订 | 定期更新测试集，测试时效性 |
| 术语专业化 | 民航专业术语 + 法律术语 | 测试术语解释能力 |
| 跨领域交叉 | 民航法 + 合同法 + 侵权法 | 测试跨领域推理能力 |

## 8. 推荐落地方案

### 8.1 短期 (1-2周): 补全核心评测维度

**优先级P0**:

1. **完整性评测 (Completeness)**
   - 采用 LeMAJ 的 LDP 方法
   - 在现有 claim-level 分解基础上增加 `<missing>` 标签
   - 计算 Recall = #Correct / (#Correct + #Missing)
   - 实现: 改造现有 faithfulness 评测脚本

2. **可回答性评测 (Answerability)**
   - 构建混合测试集: 70% 可回答 + 30% 不可回答/超范围
   - 计算: Refusal Precision, Refusal Recall, Answer F1
   - 实现: 扩展现有30题测试集

3. **检索 vs 生成分层评测**
   - 固定检索结果，只评测生成
   - 固定生成模型，只评测检索
   - 实现: 复用RAGAS指标或自行实现

### 8.2 中期 (2-4周): 场景化测试集构建

**优先级P1**:

4. **按IRAC框架设计场景题**
   - Issue: 航空事故争议点识别
   - Rule: 适用法规检索
   - Application: 法规适用于具体案例
   - Conclusion: 法律结论

5. **原子化rubric评估标准**
   - 参照Harvey LAB: 每个评估项是二元的pass/fail
   - 每个场景题设计5-10个原子评估标准
   - 涵盖: 事实、法律引用、严重性、建议、遗漏

6. **对抗性测试子集**
   - 10-15题覆盖: 过时法规、矛盾法规、陷阱问题、超范围、部分覆盖
   - 每题标注期望行为

### 8.3 长期 (1-2月): 全面评测体系

**优先级P2**:

7. **LLM-as-Judge 自动化评测管线**
   - 使用强模型作为judge
   - 定期用人类专家校准
   - 计算标注者间一致性 (Krippendorff's alpha)

8. **语义测试覆盖率**
   - 将知识库聚类为语义主题
   - 测量测试集对知识库的覆盖
   - 自动发现盲区并建议新测试题

9. **参考Stanford质量评价体系**
   - 在faithfulness/completeness之上增加:
   - Actionability (可操作性)
   - Empowerment (赋能性)
   - Strategic Caution (战略审慎性)
