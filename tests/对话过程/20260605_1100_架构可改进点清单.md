# 智能民航法律问答系统 — 架构可改进点清单

> 基于 codegraph 静态审查 SynthesisAgent / TreeRetriever / CitationAgent / LegalOrchestrator._answer / empty evidence 兜底链路
> 优化目标：Faithfulness ↑、hallucinated ↓、端到端延迟 ↓
> 不动：Faithfulness 评测体系、CE supported 率不再作为质量指标

---

## 一、代码现状关键事实（codegraph 静态审查）

### 1.1 SynthesisAgent（legalbot/agents.py:765-984）

**8 条规则的潜在问题：**
- Rule 4（"宁可多标合理推断/一般建议"）和 Rule 7/8（"覆盖度要求：并列子项各列一条，列举要分别列条"）方向冲突
  - Rule 4 鼓励 LLM 在不确定时**降低 source 等级**
  - Rule 7/8 鼓励 LLM **拆细 claim**——每多一条 sub-claim，unverifiable 风险 ×N
  - 实战结果：30 题里 6 题（20%）"看似专业但实际不可靠"——大概率是 Rule 7/8 的副作用
- `EVIDENCE_SORT_BY_CE`（line 773-774）：把 supported evidence 排前
  - `EVIDENCE_SORT_MIN_SUPPORTED_CONF=0.5` 把"低置信度 supported"降级为 partial
  - 但 `CROSS_ENCODER_CITATION_THRESHOLD=0.25` 决定了"什么算 supported"——0.26 的 evidence 在 CE 视角是 supported（合法），在合成器视角是 partial（被排到后面）
  - **双口径不一致**：CE 把 0.26 当 supported，排序把它当 partial，逻辑矛盾

**兜底链路 gap：**
- `_render_evidence_fallback`（line 848-874）触发条件：`evidence and self._looks_like_refusal(answer)`
- LLM 给"基于通用法律知识"的兜底（非明显拒答）→ 兜底链路跳过 → **把"看似专业的 unverifiable 答案"直接吐给用户**
- evidence=[] 时整个 if 跳过：见 §5

### 1.2 TreeRetriever（legalbot/tree_retrieval.py:101-485）

**Level 0→1 prior 是有效的：**
- `TREE_LAW_PRIOR_WEIGHT=0.2` 把父法规分数线性映射到章节分数：`chapter_score += 0.2 * (law_score - min) / range`
- 实验配比显示这是有效的"信号桥梁"——但 `TREE_ADAPTIVE_K_ENABLED=False` 让其变成纯截断

**Level 0→2 prior 缺失：**
- article_score 只看 art_vec + art_bm25 融合（line 290-297）
- 与 chapter_score / law_score 完全脱钩
- 即便"父章节置信度很高"也不能帮助 article 排序

**Level 2 数量设置冗余：**
- `expanded_ch = min(top_chapters * 2, TREE_CHAPTER_EXPANSION=30) = 30`
- `top_chapters = TREE_TOP_CHAPTERS = 30`
- 2x 扩展后还是 30 上限——**2x 表达式在当前配置下无意义**
- `article_candidate_count = min(top_k * 4, TREE_ARTICLE_CANDIDATES=30) = 30`（top_k=10）
- 实际 Level 2 候选 = 30，最终 top-k=10

**orphan 文章补救：**
- `TREE_ORPHAN_TOP_LAWS=5`：把前 5 个法规的"无章节文章"补进候选
- 但 `TREE_CHAPTER_PER_LAW=10`：每法规最多 10 章节，**章节先被限流**，orphan 补救未必有效

**RRF_K=15 偏大：**
- RRF 加权 = 1/(k+rank)，k=15 意味着 rank 1 vs rank 10 差距很小（1/16 vs 1/25 = 1.6x）
- 适合"信号很弱希望靠多源融合"场景，但 30 题里常出现"1 个真答案 + 9 个总则"——需要更陡峭的 rank 惩罚

### 1.3 CitationAgent（legalbot/agents.py:598-619, legalbot/citation.py）

**CE supported 门槛过低：**
- `CROSS_ENCODER_CITATION_THRESHOLD = 0.25`（config.py:63）
- 任何 score >= 0.25 的 evidence 都标 supported
- BGE-reranker-v2-m3 在中文语料上 score 范围 0-1，0.25 在多数实验中对应"前 50% 相关"
- 与 `EVIDENCE_SORT_MIN_SUPPORTED_CONF=0.5` 不对齐

**CLAIM_LEVEL_CITATION=False：**
- 走 `_verify_evidence_level`（旧方向）：每条 evidence 找最佳 claim 匹配
- 失败时 fallback：claim_score 过低时用 question_score 替代
- **问题：fallback 后 best_score 用 `q_s > best_score` 判定更新，最佳可能变成"问题相关但 claim 不相关"的 evidence**
- `extract_claims` 失败时 `return [question]`——**所有 evidence 都和"原始问题"匹配**，退化为 question→evidence 排序

**CLAIM_LEVEL_CITATION=True（消融测试）：**
- 走 `_verify_claim_nli`：每个 claim 找最佳 evidence
- **每条 evidence 都被忽略**——1 个 evidence 对 claim-A score=0.8、对 claim-B score=0.3，只看 0.8
- 高频出现的"通用模板"evidence（依据/变更/目的）容易霸榜

**Set-Membership 是软警告：**
- line 1221-1236：`_sm_warning=True` 后只在渲染时加"⚠️待核实"
- 不删除 claim、不剥离引用、不降级 source
- **用户看到的是"看似有引用但其实引用错"的答案**

**`extract_claims` 永远跑：**
- 即便 `CROSS_ENCODER_CITATION=True`（默认），也先 LLM 抽 claim
- CE 模式下其实不需要 claim 列表（用 question 当 query 即可）
- 每次 verify 多 1 个 LLM 调用

### 1.4 LegalOrchestrator._answer（legalbot/agents.py:1087-1479）

**5-7 个 LLM 调用：**
1. `SubjectAgent.extract_subjects`（line 1095）
2. `IssueAgent.identify_issues`（line 1108）
3. `RewriteAgent.rewrite_queries`（line 1144）
4. `DecompositionAgent.decompose`（line 1192，有条件）
5. `CitationVerifier.extract_claims`（line 1210→citation.py:36，**永远跑**）
6. `SynthesisAgent._assess_relevance`（line 786，有条件）
7. `SynthesisAgent._compose_structured`（line 946）
8. `ReflexionAgent.evaluate`（line 1386，有条件）

**冗余 LLM 调用：**
- SubjectAgent + IssueAgent 看同一个 question，输出 JSON 字段互补——**可合并**
- RewriteAgent 的 `law_hints` vs IssueAgent 的 `applicable_law_domains` 是同一信息
- CE 模式下的 `extract_claims` 不需要——**可删**

**没有 pre-evidence hard gate：**
- 5 个 LLM 调用后才进 retrieve
- 如果证据为空（用户问冷门问题），5 个调用白费
- D 类 number guard、`SYNTHESIS_REFUSAL_FALLBACK`、`RELEVANCE_GATE` 都在 evidence 必真时触发——evidence=[] 时全部跳过

**Reflexion 循环：**
- `MAX_REFLEXION_ITERATIONS=2`
- 每轮 1 evaluate + 1 compose = 2 LLM/轮
- 最坏情况 4 额外 LLM 调用
- 快速路径：`all(c.status=='supported' and c.confidence>=0.7)` 才跳过
- 当前 baseline CE supported 率 51%，但**部分高 confidence**——不是全 0.7+ 才走快速路径

**EVIDENCE_LOAD_LIMIT 与 SYNTHESIS_EVIDENCE_LIMIT 都是 12：**
- config.py:92-93：两个常量同值，没必要分开
- 实际 LLM 看到 12 条 evidence，token 压力较大（每条 ≤ 2000 字符 = 24k 字符 ≈ 8k tokens）

### 1.5 evidence=[] 时的兜底链路（最大风险）

**完整调用路径：**
```
compose_answer (line 769)
  ├─ if RELEVANCE_GATE_ENABLED and evidence:  ← evidence 必真才进
  │     assessment = self._assess_relevance(...)
  ├─ elif SYNTHESIS_JSON_MODE:
  │     answer = self._compose_structured(question, plan, evidence=[], evidence_text=[], ...)
  └─ else:
        answer = self._compose_free_text(question, plan, evidence_text=[], ...)
  └─ if SYNTHESIS_REFUSAL_FALLBACK and evidence:  ← evidence 必真才进
        fallback = self._render_evidence_fallback(...)
```

**问题清单：**
- relevance gate 跳过：evidence=[] 时直接进合成
- `_compose_structured`：evidence_index = ""（空字符串），user message 末尾"请先通读所有【证据全文】"——**没有 evidence 也通读**
- `_compose_free_text`：直接说"已读取证据:"**然后什么都没有**——LLM 自由发挥
- `SYNTHESIS_REFUSAL_FALLBACK` 整个 if 跳过
- D 类 number guard：`if NUMBER_GUARD_ENABLED and state.evidence`——跳过
- **核心：LLM 拿到空 evidence + "必须只基于证据回答"提示 = hallucination 邀请**

## 二、可改进点清单（按 ROI 排序）

### P0 - 必改（影响 Faithfulness 指标）

#### P0-1. 加 empty evidence hard gate
- **位置**：`legalbot/agents.py:769 compose_answer` 开头
- **改动**：
  ```python
  def compose_answer(self, question, plan, evidence, citations, conflicts):
      if not evidence:
          return ("【结论】\n抱歉，未检索到相关法律条文。\n\n"
                  "【建议】\n建议您咨询专业法律人士或提供更具体信息。")
      # ... 现有逻辑
  ```
- **预期影响**：unverifiable 率 ↓ 5-10pp，hallucinated 率 ↓ 3-5pp
- **验证**：test_30questions 中挑 evidence 必然为空的冷门问题，确认触发硬拒答

#### P0-2. Set-Membership 失败时强制 source 降级
- **位置**：`legalbot/agents.py:1221-1236` Set-Membership 段
- **改动**：
  ```python
  # 旧：_sm_warning=True
  # 新：source 降级 + 移到"补充说明"区
  if node_ids and not any(nid in valid_node_ids for nid in node_ids):
      claim['source'] = '一般建议'  # 强制降级
      warned += 1
  # 渲染时通过 _render_structured_answer 的 source 分区自动归到【补充说明】
  ```
- **预期影响**：faithful 率 ↑ 3-5pp，partial 率 ↑，"看似有引用但引用错"问题消除
- **验证**：30 题 ablation，对比"警告保留"vs"降级到补充"

#### P0-3. 拉高 CE supported 门槛
- **位置**：`legalbot/config.py:63-64`
- **改动**：
  ```python
  CROSS_ENCODER_CITATION_THRESHOLD = 0.40   # 旧 0.25
  CROSS_ENCODER_PARTIAL_THRESHOLD = 0.20    # 旧 0.15
  ```
- **预期影响**：CE "假支持"率 ↓，EVIDENCE_SORT 不再被"勉强通过"污染
- **验证**：30 题，统计支持/部分支持占比 + faithful 率变化

### P1 - 下迭代改（性能/可维护性）

#### P1-1. 合并 SubjectAgent + IssueAgent
- **位置**：`legalbot/agents.py:152-254` 两类合并
- **新 schema**：
  ```json
  {
    "subjects": {...},
    "events": [...],
    "relationships": [...],
    "legal_issues": [...],
    "issue_types": [...],
    "applicable_law_domains": [...],
    "uncertain_facts": [...],
    "assumptions": [...],
    "alternative_paths": [...],
    "clarification_policy": {...}
  }
  ```
- **预期影响**：节省 1 LLM 调用，端到端延迟 ↓ 10-15%

#### P1-2. CE 模式下删除 extract_claims 调用
- **位置**：`legalbot/citation.py:39-47 verify_with_cross_encoder`
- **改动**：
  ```python
  def verify_with_cross_encoder(self, question, legal_issues, evidence):
      if not evidence: return []
      if CLAIM_LEVEL_CITATION:
          return self._verify_claim_nli(question, legal_issues, evidence)
      # CE 方向不需要 claim 列表，用 legal_issues 列表当 query
      queries = legal_issues if legal_issues else [question]
      return self._verify_evidence_level_ce(queries, evidence)
  ```
- **预期影响**：节省 1 LLM 调用/题

#### P1-3. pre-evidence 评估门控
- **位置**：`legalbot/agents.py:_answer` 在 subject+issue 之后、rewrite 之前
- **改动**：加一次"问题是否可由民航法回答"的 LLM 评估（用 IssueAgent 输出版本扩展一个 `answerable: bool` 字段）
- **预期影响**：冷门问题（"今天天气""Python 怎么用"）直接兜底，不进检索，**节省 3-4 LLM 调用**

#### P1-4. 统一 EVIDENCE_LOAD_LIMIT 与 SYNTHESIS_EVIDENCE_LIMIT 语义
- **位置**：`legalbot/config.py:92-93`
- **改动**：
  ```python
  EVIDENCE_LOAD_LIMIT = 12      # 检索回读
  SYNTHESIS_EVIDENCE_LIMIT = 8  # 合成时只看前 8（节省 token）
  ```
- **预期影响**：合成 LLM token ↓ 33%，faithful 率可能反而 ↑（fewer distractors）

### P2 - 长期优化

#### P2-1. Rule 7/8 收紧
- **位置**：`legalbot/agents.py:_compose_structured` Rule 7/8
- **新规则**：
  ```
  7. 并列子项不超过 3 个时，每种独立列条；超过 3 个时，evidence 完整支持多少列多少，其余归 source="一般建议"。
  8. evidence 文本中读不到的"分类"（如"5 种情形"）用合并表达"主要包括以下 X 类"，不强行分条。
  ```
- **预期影响**：unverifiable 率 ↓，partial 率 ↑

#### P2-2. Level 0→2 prior 传播
- **位置**：`legalbot/tree_retrieval.py:295-297`
- **改动**：在 art_rrf 融合后加 `prior_w_2 = 0.1 * chapter_score` 信号
- **预期影响**：article 排序与 chapter 信号耦合，避免"高分章节下挂低分文章"

#### P2-3. 调整 RRF_K
- **位置**：`legalbot/config.py:47`
- **改动**：`RRF_K = 10`（旧 15）
- **预期影响**：rank 差异更陡峭，"真答案"浮上来

#### P2-4. Evidence 加载阶段的早期拒答
- **位置**：`legalbot/agents.py:RetrievalAgent.retrieve` 末尾
- **改动**：如果 EVIDENCE_LOAD_LIMIT=12 但实际仅 1-2 条 evidence 且全部 score < 0.3，**直接 return []**
- **预期影响**：让 hard gate 触发更频繁，避免低质量 evidence 撑场

4. **KEYWORD_ROUTING_ENABLED**：v3 +18pp，**保持**
5. **CLAIM_LEVEL_CITATION=False**：消融测试"新方向太激进"，**保持旧方向**
6. **CROSS_ENCODER_CITATION=True**：替代 LLM 校验，节省 token，**保持**
7. **NUMBER_GUARD_ENABLED**：D 类防御已有效，**保持**

## 四、推荐改造顺序

1. **P0-1**（empty evidence hard gate）— 单点改动，立即降低 hallucinated 率
2. **P0-3**（CE 门槛 0.25→0.40）— 单点改动，1 行配置
3. **P0-2**（Set-Membership 降级）— 局部代码改动，5-10 行
4. **P1-4**（evidence limit 拆分）— 1 行配置

6. **P1-1**（SubjectAgent + IssueAgent 合并）— 局部重构，需要新 schema
7. **P2-1**（Rule 7/8 收紧）— 文本调整
8. **P1-3**（pre-evidence 评估门控）— 较大的流程改动
9. **P2-2/3/4**（树检索调优）— 需要 ablation 验证

每步改造后跑 `tests/test_faithfulness.py` + `tests/test_summary.py` 三层评测验证。

## 五、关键风险

- **P0-1 改 hard gate 后**：原 baseline 的 1 道"拒答题"（FAITHFULNESS 6/30 5%）会变成"硬拒答"，统计上影响 1-2pp。需要同步看 F1 变化
- **P0-3 拉高 CE 门槛后**：支持率从 51% 跌到 ~30%（消融预测），但 faithful 率应 ↑，"假支持"消失
- **P1-1 合并 Agent 后**：context 窗口翻倍，LLM JSON 解析失败率可能 ↑，需要重试机制
- **P1-3 pre-evidence 评估**：新 LLM 调用会引入延迟，如果不开 cache 净亏损
