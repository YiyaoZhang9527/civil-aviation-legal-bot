# LLM 解决路由表质量 — 前沿方案调研

**时间:** 2026-06-01
**问题:** 手写关键词路由表质量差，如何用 LLM 自动化解决？

## 与我们项目最相关的论文（按相关性排序）

### 1. RealRoute: 动态查询路由 — Retrieve-then-Verify（最相关）

**论文:** "Dynamic Query Routing System via Retrieve-then-Verify Paradigm" (arXiv 2604.20860, 2026-03)
**URL:** https://arxiv.org/html/2604.20860v1

**核心思路：** 不预建路由表，而是先在所有知识库检索一遍，然后用验证链检查检索结果质量，动态决定是否需要重新路由到其他知识库。

**对我们的价值：** 这正是我们的场景 — 129 部法律作为独立知识库，query 需要路由到正确的法规。RealRoute 的 verify 步骤可以用 Cross-Encoder 实现：如果第一轮检索结果的精排分数都低于阈值，说明路由错了，触发重新路由。

### 2. Adaptive Query Routing: 分层混合检索（直接对标）

**论文:** "Adaptive Query Routing: A Tier-Based Framework for Hybrid Retrieval Across Financial, Legal, and Medical Documents" (arXiv 2604.14222, 2026-04)
**URL:** https://arxiv.org/pdf/2604.14222

**核心思路：** 用 GPT-4o-mini 做查询分类器，将 query 分到 4 个复杂度层级，不同层级用不同检索策略：
- Tier 1 (简单查找) → 向量检索
- Tier 2-3 (多段落/交叉引用) → 树推理
- Tier 4 (多文档综合) → 混合检索

**关键发现：**
- 法律领域：Tree Reasoning 得分 0.883 > Vector RAG 0.800 > Hybrid 0.850
- 医疗领域：Tree Reasoning 0.933 远超其他
- 金融领域：Vector RAG 0.910 领先
- **交叉引用召回：Tree 100% vs Vector 91.7%**

**对我们的价值：** 我们已经有 Tree Reasoning（法→章→条逐层剪枝），这篇论文验证了这个架构在法律领域的优越性。路由不应该砍掉法规，而应该决定用哪条检索路径。

### 3. R1-Router: 多知识库动态路由（思路启发）

**论文:** "Learning to Route Queries across Knowledge Bases for Step-wise Retrieval-Augmented Reasoning" (arXiv 2505.22095, 2025-05)
**URL:** https://arxiv.org/html/2505.22095v1

**核心思路：** MLLM 在推理过程中动态决定"什么时候查"和"查哪个知识库"。用强化学习（Step-GRPO）训练模型学会路由，不需要手写规则。

**关键：** 路由不是一次性决策，而是推理过程中逐步决定的 — 可能先查一个知识库，发现不够，再查另一个。

**对我们的价值：** 我们的 reflexion agent 已经有"反思"机制，可以在第一轮检索不充分时动态换法规重查。但当前实现还没做到跨法规重路由。

### 4. SPLADE: 学习型稀疏检索（替代路由表的方案）

**论文/资源:**
- Pinecone SPLADE 教程: https://www.pinecone.io/learn/splade/
- Hybrid Search (BM25/SPLADE + Dense + RRF): https://blog.gopenai.com/hybrid-search-in-rag-dense-sparse-bm25-splade-reciprocal-rank-fusion-and-when-to-use-which-fafe4fd6156e

**核心思路：** SPLADE 是 transformer 模型，将文本映射到稀疏向量空间（30000+ 维），能自动做**词扩展**：
- 输入 "car" → 自动关联 "vehicle, automobile, automotive"
- 不需要手写同义词表

**关键对比：**
| 方法 | 优势 | 劣势 |
|------|------|------|
| BM25 | 快，支持 OOV 词 | 无语义理解 |
| SPLADE | 自动同义词扩展，可解释 | 推理慢 100-300ms |
| Dense | 语义匹配最强 | 不可解释 |

**实用建议（来自 GoPenAI 文章）：**
- 小语料库（100-300 文档）用 RRF k=10-20，不要用 k=60
- 先用 BM25 + Dense + RRF，验证有效后再考虑 SPLADE
- SPLADE 可以离线预计算文档向量，只在查询时跑一次推理

**对我们的价值：** 我们已经有 `google-bert/bert-base-chinese` 做 SPLADE 稀疏编码。问题是当前稀疏检索只在条级（Level 2），没有用在法级路由（Level 0）。如果 Level 0 也用 SPLADE，就不需要手写关键词路由了。

### 5. DRAGIN: 基于 LLM 注意力的动态检索（新思路）

**论文:** "DRAGIN: Dynamic Retrieval Augmented Generation based on the Information Needs of LLMs" (ACL 2024)
**URL:** https://aclanthology.org/anthology-files/pdf/acl/2024.acl-long.702.pdf

**核心思路：** 利用 LLM 的 self-attention 权重来判断"模型现在需要什么信息"，然后构造精确的检索 query。不是人写关键词，而是让模型自己表达信息需求。

**对我们的价值：** 可以用 LLM 的 attention 来自动提取 query 中的关键概念，替代手写路由表。但这需要访问 LLM 内部状态，API 模型做不到。

## 实施建议（按难度排序）

### 短期（1-2天）：LLM 离线生成路由表 v2

当前 `law_routes.json` 质量差（缺少"备降"、"燃油"等核心词）。用 LLM 重新生成：
1. 把每个法规的全文（或 summary）喂给 LLM
2. 让 LLM 列出"用户可能用哪些词/短语来问到这部法规"
3. 输出包含用户自然语言的关键词，而不是法规官方标题

这其实就是当前 `_load_law_routes()` 的设计意图，只是生成质量太差。

### 中期（1周）：SPLADE 替代关键词路由

Level 0 法级检索目前用 Dense + BM25 + RRF。如果加上 SPLADE 稀疏向量：
- "备降" 会自动扩展到相关法规的语义空间
- 不需要手写任何关键词规则
- 我们已有 `google-bert/bert-base-chinese` 模型，只需在 Level 0 也跑 SPLADE

### 长期（2-4周）：Retrieve-then-Verify 动态路由

借鉴 RealRoute：
1. 第一轮在所有法规里粗检索
2. Cross-Encoder 验证结果质量
3. 如果验证不通过，动态换法规重检索
4. 这需要在 reflexion agent 里加入"跨法规重路由"能力
