## 二、行业三大解法

### 解法 1：检索层 — 从"文档级"到"段落级"

**Remedy Legal（2026.03）的实践：82% → 99.8%**

核心改动：
1. **分块（Chunking）**：把 85,000 token 的判例文书切成 ~250 token 的段落。每个段落单独做 embedding，代表一个聚焦的法律观点，而不是整篇文书的"重心"
2. **混合搜索**：向量搜索擅长语义匹配（"安静享用"→"和平占有"），但对法规编号（"第121条"、"CCAR-92"）完全盲区。加 BM25 关键词搜索后，用 RRF 融合两个排名
3. **段落→文档回溯**：匹配到段落后，追溯到父文档获取完整上下文

**对我们意味着什么：**
- 我们已经有 BM25 + 向量 + hint 的混合搜索和 RRF 融合
- 但我们缺的是**段落级分块** — 当前是按条文（article）整条做检索单元，长条文（1500+ 字）的 embedding 被"重心化"稀释
- 可行改进：对长条文做子段落切分，检索时用段落 embedding，回溯时用整条文上下文

**德国民法典分块研究（arXiv 2605.19806, 2025）：**
- 测试了固定大小、语义切分、法律结构切分（按条文/款/项）等策略
- 结论：**法律结构切分 > 固定大小切分 > 语义切分**。法律文本有天然的结构边界（条、款、项），应该利用这个结构而不是暴力切割
- Summary-Augmented Chunking (SAC)：给每个分块生成一个摘要，用摘要做检索初筛，用原文做精排。在 LegalBench-RAG 上效果最好

### 解法 2：生成层 — Self-RAG / Corrective-RAG / 结构化输出

**Self-RAG（Asai et al., 2024）：**
- LLM 在生成过程中自我反思，插入"反思 token"：`[Retrieve]`（需要检索吗？）、`[ISREL]`（检索结果相关吗？）、`[ISSUP]`（有证据支撑吗？）
- 生成每个 claim 后自动校验是否被证据支撑，不支撑就重新检索
- 缺点：需要专门微调的模型，不能直接用 API

**Corrective-RAG（CRAG, Yan et al., 2024）：**
- 轻量版 Self-RAG：检索后先评估检索质量（用 LLM 打分）
- 如果检索质量低，触发纠错：改写 query 重新检索、或用 web search 补充
- 不需要微调模型，可以在任何 LLM API 上实现

**结构化输出 + 引用接地（Citation Grounding）：**
- 强制 LLM 输出结构化 JSON：每个 claim 必须附带 evidence_id / node_id
- 没有 evidence_id 的 claim 直接拒绝
- 这是 Lexis+ AI 和 Westlaw Precision 的核心做法 — **"不引用不输出"**
- 我们已经在做（SYNTHESIS_JSON_MODE + SET_MEMBERSHIP_CHECK）

**对我们意味着什么：**
- Self-RAG 太重（需要微调模型），不适用
- **CRAG 可以做**：检索后用 cross-encoder 评估质量，低质量时触发 query 改写或扩展检索。我们的 reflexion 循环已经在做类似的事，但只检查答案质量不检查检索质量
- 结构化输出我们已实现，继续保持

### 解法 3：验证层 — 多重交叉校验

**Stanford HAI 的四类法律 AI 错误分类：**
1. **捏造（Fabrication）**：完全虚构的案例/法条
2. **误述（Mischaracterization）**：真实法条但扭曲了含义
3. **谄媚（Sycophancy）**：用户说错，AI 顺着错
4. **错误权威（Incorrect Authority）**：引用了不相关的法规

**Lexis+ AI / Westlaw 的做法：**
- **Citator 系统**：每条引用都经过预构建的引用关系图验证 — 该法条是否仍然有效？是否被后续修订？是否有判例引用？
- **Source-restricted RAG**：只在经过验证的数据库中检索，不允许 LLM 超出数据源范围
- **拒绝机制**：当置信度不够时，工具会主动说"我没有足够信息"，而不是编造

**学术界的做法（Fine-grained Claim-level RAG, arXiv 2605.21071）：**
- 把答案拆解到 claim 级别，逐条验证
- 每个 claim 独立判断：supported / unsupported / conflicting
- 用 NLI（自然语言推理）模型做 claim-evidence 匹配
- **这正是我们 ClaimNLI 在做的事**，但我们用了 cross-encoder 代替专用 NLI 模型

**对我们意味着什么：**
- Citator 太重（需要构建全法条引用关系图），但可以做一个轻量版：法条是否被后续修订
- **拒绝机制很重要**：我们当前所有 claim 都会尝试回答，应该在证据不足时主动拒绝
- Claim-level 验证方向正确，但 cross-encoder 对声明性文本打分偏低（我们已发现的问题），需要用原始问题回退打分

## 三、优先级排序（按投入产出比）

| 优先级 | 改进 | 预期收益 | 实现难度 | 参考 |
|--------|------|---------|---------|------|
| **P0** | 条文内子段落分块 | 检索精度 +10-15% | 中 | Remedy Legal, 德国民法典研究 |
| **P1** | 检索质量评估（CRAG） | 减少无效检索轮次 | 低 | Corrective-RAG |
| **P1** | 证据不足时拒绝回答 | 幻觉率 -5-10% | 低 | Lexis+ AI |
| **P2** | SAC：摘要辅助检索 | 长文档召回率 +5-10% | 中 | arXiv 2510.06999 |
| **P2** | 法条有效性标注（轻量 Citator） | 过期法条误引 -3-5% | 高 | Westlaw Citator |
| **P3** | Self-RAG（需微调模型） | 全面提升 | 很高 | Self-RAG 论文 |

## 四、我们的优势

对比行业，我们已经在做对的事情：
- ✅ 混合搜索（BM25 + 向量 + hint + RRF）
- ✅ Cross-encoder 精排
- ✅ 结构化 JSON 输出 + Set-Membership 校验
- ✅ Claim-level 验证
- ✅ 自检循环（Reflexion）
- ✅ 问题回退打分

还没做的关键项：
- ❌ 子段落分块（条文内部再切分）
- ❌ 检索质量评估（检索后先评估再决定是否继续）
- ❌ 主动拒绝（证据不足时说"我不知道"）
- ❌ 摘要辅助检索（SAC）
