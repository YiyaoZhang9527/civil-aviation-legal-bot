# P2 调研：多 query 合并冲淡 + 答案引用格式不稳定

日期：2026-06-01

## 问题A：多 query 合并冲淡

### 现象
Q14 无人机安全距离：单 query 全部命中 CCAR-92，LLM 改写 query 检索到空管规则，合并后正确证据被稀释，supported 仅 2/12。

### 根因
等权合并策略。改写 query 语义偏移时，无关文档以同等资格参与排序。

---

### 方案 A1：加权倒数排名融合（WRRF）⭐ 推荐

**原理**：标准 RRF `Score(d) = Σ 1/(k + rank_i(d))` 所有 query 等权。WRRF 加入 query 级权重：`Score(d) = Σ w_i/(k + rank_i(d))`。原始 query w=1.0，改写 query w=0.3~0.5。

**实现**：
1. 每个 query 分配固定权重：原始 w=1.0，改写 w=0.4
2. 各 query 独立检索 top-K，记录 rank 和来源权重
3. 计算 WRRF 分数排序取 top-8
4. 可选：检索相似度 min-max 归一化作为置信度乘入权重（CCNC 2026 做法）

**评估**：实现简单，仅改合并层。CCNC 2026 论文实测 WRRF 全面优于标准 RRF。Elasticsearch 8.x 已原生支持。**推荐度：高**。

---

### 方案 A2：来源法规分组去重（Source-Aware Grouping）

**原理**：合并后按来源法规分组，每个法规最多保留 N 篇（如 N=3），防止单法规或无关法规占过多席位。

**实现**：
1. 文档元数据需要 `source_regulation` 字段（已有 `metadata.source`）
2. 合并排序后遍历，维护法规计数器，单法规最多 3 篇
3. 额外：原始 query 排名第一的法规文档给额外加分（来源锚定）

**评估**：与 WRRF 正交可叠加。**推荐度：中高**。

---

### 方案 A3：原始 query 优先保留（Primary Query Guarantee）

**原理**：原始 query 检索 top-4 直通最终列表，剩余 4 位从改写 query 结果中补充。

**实现**：原始 query top-4 无条件进入，改写结果去重后按分数填充。

**评估**：5 行代码，但牺牲改写 query 发现遗漏证据的能力。适合做 A1 安全网。**推荐度：中**。

---

### 问题A 推荐组合

**A1（WRRF）+ A2（来源分组）+ A3（保底兜底）**

1. 原始 query（w=1.0）保底 top-4 直通
2. 改写 query 按 WRRF（w=0.4）计算分数
3. 合并后按法规来源分组，单法规最多 3 篇
4. 最终取 top-8

预期：CCAR-92 文档获得高权重+保底，空管规则被权重和分组双重限制。

## 问题B：答案引用格式不稳定

### 现象
LLM 答案中法条引用格式不统一，引用校验无法覆盖所有声明，text_refs 多为空。

### 根因
LLM 自由文本输出靠 prompt 约束格式但不可靠，校验侧正则无法覆盖所有写法。

---

### 方案 B1：结构化输出强制引用格式（Structured Output）⭐ 推荐

**原理**：用 LLM API 的 structured output / JSON mode，强制按 schema 输出。答案拆分为 claims 数组，每个 claim 必须带 source_ref。

**实现**：
```python
class Claim(BaseModel):
    text: str
    law_name: str          # "CCAR-92"
    article_number: str    # "第92.51条"

class LegalAnswer(BaseModel):
    summary: str
    claims: list[Claim]
```
LLM 调用时用 `response_format` 强制 JSON，校验直接读结构化字段。

**评估**：OpenAI/Anthropic/Gemini 均支持。格式 100% 可控。延迟实测无差异。后处理拼接为自然文本即可。**推荐度：高**。

---

### 方案 B2：引用后处理标准化（Post-Processing Normalization）

**原理**：LLM 输出自由文本后，用正则做引用格式标准化。

**实现**：建立引用模式库（`《法名》第X条`、`第X条` 等），正则扫描替换，无法规名的用上下文最近法规名补全。

**评估**：正则覆盖永远不完整。适合做 B1 的补充兜底。**推荐度：中**。

---

### 方案 B3：Prompt 强化引用格式

**原理**：System prompt 规定严格引用格式 + few-shot 示例。

**评估**：成本最低但 LLM 不保证遵守，长答案后段易遗忘。适合与 B1 配合。**推荐度：中高（配合 B1）**。

---

### 方案 B4：CitaLaw 三段论评估（评估层升级）

**原理**：arXiv 2412.14556（ACL 2025 Findings）提出：法规条文→案件事实→回答 的三段论验证，评估引用是否在逻辑上支持回答。

**评估**：提升评估质量，但不解决生成层格式问题。适合后续评估升级。**推荐度：中高**。

---

### 问题B 推荐组合

**B1（结构化输出）+ B3（Prompt 强化）**

1. Structured output 强制 LLM 按 schema 输出（claims + source_ref）
2. Prompt 附带 few-shot 提升引用内容准确性
3. 后处理将结构化输出拼接为自然文本

预期：text_refs 覆盖率从低位提升至接近 100%。

---

## 实施优先级

| 优先级 | 方案 | 改动范围 | 预期收益 |
|--------|------|----------|----------|
| P0 | A1 WRRF 加权合并 | 合并层 | 直接解决冲淡 |
| P0 | B1 结构化输出 | LLM 调用层 | 直接解决格式 |
| P1 | A3 原始 query 保底 | 合并层 | 安全网 |
| P1 | A2 来源分组 | 合并层 | 多法规覆盖 |
| P2 | B3 Prompt 强化 | Prompt 层 | 引用准确性 |
| P2 | B2 后处理标准化 | 校验层 | 兜底 |
