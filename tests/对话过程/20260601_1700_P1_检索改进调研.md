# P1 检索改进调研：跨法规检索 & Level 1 噪声控制

日期：2026-06-01
背景：129部法规三级树检索，3道旅客服务题跨法规检索失败，Level 1 扩展引入噪声

---

## 一、问题A：跨法规检索失败 — 成熟方案

### 方案A1：查询分解（Query Decomposition）+ 多轮检索

- **名称**: Query Decomposition + Multi-Hop Retrieval
- **原理**: LLM 将复合问题拆解为子问题，每个子问题独立检索，最后合并证据
- **实现方式**:
  1. Planner 节点：LLM 将"航班延误餐饮"拆为 ["航班延误规定", "餐饮服务标准"]
  2. Executor 逐个执行子问题检索，收集答案
  3. Final 节点合并所有证据生成最终答案
- **延迟影响**: 每个子问题 +1 次检索 + 1 次 LLM 调用。2-3 子问题约增加 2-3s
- **适用性**: ★★★★★ 直接命中"跨法规拼图"需求
- **来源**: [Multi-hop RAG Guide](https://www.learnwithparam.com/blog/multi-hop-rag-query-decomposition) | [EfficientRAG (EMNLP 2024)](https://aclanthology.org/2024.emnlp-main.199.pdf)

### 方案A2：EfficientRAG — 无 LLM 的迭代查询生成

- **名称**: EfficientRAG
- **原理**: 用轻量 Token Tagger（非 LLM）标记已找到/缺失的信息，用 Label Token Generator 生成下一轮查询，无需每轮调用 LLM
- **实现方式**:
  1. 初始检索 → Token Tagger 判断哪些 token 已覆盖/仍缺失
  2. Label Token Generator 基于缺失 token 生成新查询
  3. 迭代直到无新信息或达到轮次上限
  4. Filter 模块过滤无关文档
- **延迟影响**: 比标准 multi-hop 快 3-5x（无需每轮 LLM），约增加 0.5-1s
- **适用性**: ★★★★☆ 实现复杂度中等，但无需 LLM 调用是巨大优势
- **来源**: [EfficientRAG Paper (EMNLP 2024)](https://aclanthology.org/2024.emnlp-main.199.pdf)

### 方案A3：多图多 Agent 递归检索

- **名称**: Multi-Graph Multi-Agent Recursive Retrieval
- **原理**: 构建法规层次图 + 定义图，多 Agent 协作：初始检索 Agent → 定义查询 Agent → 路由 Agent → 递归检索 Agent → 回答 Agent
- **实现方式**:
  1. 构建法规条款交叉引用图（条款级节点 + 引用边）
  2. 构建法律术语定义图
  3. LangGraph 编排多 Agent 工作流，Router 检测是否需跨文档引用
- **延迟影响**: 多 Agent 编排增加 3-8s，适合离线/非实时场景
- **适用性**: ★★★☆☆ 过于重量级，适合超长法规链式引用场景，我们的 129 部法规场景杀鸡用牛刀
- **来源**: [Legal Document RAG (Medium)](https://medium.com/enterprise-rag/legal-document-rag-multi-graph-multi-agent-recursive-retrieval-through-legal-clauses-c90e073e0052)

### 方案A4：答案规划 + 证据追踪

- **名称**: Answer Planning with Evidence Tracking
- **原理**: 生成答案前先建"答案计划"——列出要做的声明、每条声明需要的来源、还缺什么信息，然后定向补检索
- **实现方式**:
  1. 第一轮宽检索，LLM 生成答案计划（claim → source map）
  2. 标记缺失证据的 claim
  3. 对缺失 claim 生成定向查询，二次检索
  4. 合并证据生成最终答案
- **延迟影响**: +1 次 LLM 调用 + 可能的二次检索，约 +2s
- **适用性**: ★★★★☆ 与方案A1互补，可组合使用
- **来源**: [Query Orchestration for RAG](https://www.amestris.com.au/blog/rag-query-orchestration.html)

## 二、问题B：Level 1 扩展引入噪声 — 成熟方案

### 方案B1：Adaptive-k — 相似度分数落差自动截断

- **名称**: Adaptive-k Retrieval (EMNLP 2025)
- **原理**: k = argmax(s_i - s_{i+1})，找到相似度分数的最大落差点作为截断边界
- **实现方式**:
  1. Level 1 检索返回 top-N（如 N=40）候选章节
  2. 计算相邻分数差 s_i - s_{i+1}
  3. 在最大落差处截断，只保留前面的候选
  4. 最小 k 保底（如 k_min=5）防止过度截断
- **延迟影响**: ★★★★★ 零额外延迟，纯数学计算
- **适用性**: ★★★★★ 直接替换固定扩展，简单有效
- **注意**: 如果相似度分布平坦（通用章节太多），落差不明显时需设最小 k
- **来源**: [Adaptive-k (GitHub)](https://github.com/megagonlabs/adaptive-k-retrieval) | [Adaptive-k 解读](https://indranildchandra.medium.com/adaptive-k-retrieval-a-smarter-way-to-trim-context-in-rag-687a399a67fe)

### 方案B2：法规名额分配（Source Diversity Quota）

- **名称**: Source-Aware Diversity Quota
- **原理**: Level 1 候选中，每部法规最多占 N 个名额，确保法规多样性
- **实现方式**:
  1. Level 1 检索 top-40 章节
  2. 按法规分组，统计每部法规的候选数
  3. 每部法规最多保留 top-N（如 3 个）章节
  4. 剩余名额从被截断的法规中按分数补充
- **延迟影响**: 几乎为零（分组统计 + 截断，纯规则）
- **适用性**: ★★★★★ 与 Adaptive-k 互补，防止某部法规垄断候选
- **来源**: [Fairness-Aware Reranking (ACM)](https://dl.acm.org/doi/abs/10.1145/3626772.3657794) | [Fairer IR (Rutgers)](https://rucore.libraries.rutgers.edu/rutgers-lib/65375/PDF/1/play/)

### 方案B3：层次感知重排（Hierarchy-Aware Reranking）

- **名称**: ToolRerank / HRR (LREC-COLING 2024)
- **原理**: 重排时考虑层次结构，同法规章节做组内去重，跨法规章节加权提升
- **实现方式**:
  1. Level 1 检索 top-40 章节
  2. Cross-encoder 重排（已有 reranker）
  3. 层次调整：同法规内只保留最高分章节的相邻章节，跨法规章节加分
- **延迟影响**: 重排已有，层次调整为零开销
- **适用性**: ★★★★☆ 可在现有 reranker 基础上增加后处理规则
- **来源**: [ToolRerank (arXiv)](https://arxiv.org/html/2403.06551v1)

### 方案B4：MMR 多样性选择

- **名称**: Maximal Marginal Relevance (MMR)
- **原理**: 迭代选择时兼顾相关性和多样性：argmax[λ·sim(q,d) - (1-λ)·max_sim(d,D_selected)]
- **实现方式**:
  1. Level 1 候选按分数排序
  2. 贪心选择：每次选与 query 最相关且与已选集最不相似的章节
  3. λ 控制相关性 vs 多样性权衡
- **延迟影响**: O(n²) 相似度计算，对于 40 个候选可忽略
- **适用性**: ★★★★☆ 经典方法，但"多样性"不等于"法规多样性"——需改为按法规 ID 分组
- **来源**: 信息检索经典方法，1998 年 Carbonell & Goldstein

## 三、推荐方案

### 问题A推荐：查询分解（方案A1）+ 证据追踪（方案A4）

**理由**：
- 最直接解决"跨法规拼图"问题
- 实现简单：1 个 LLM 调用做分解 + 多次现有检索 + 1 次 LLM 合并
- 延迟可控：2-3 子问题约 +3s
- 无需改动索引结构

**最小实现**：
```
1. LLM 分解查询 → ["航班延误赔偿规定", "机上餐饮服务标准"]
2. 逐个子问题走现有三级树检索
3. 合并所有子问题的条级证据
4. LLM 基于合并证据生成答案
```

### 问题B推荐：Adaptive-k（方案B1）+ 法规名额分配（方案B2）

**理由**：
- Adaptive-k 零延迟开销，直接替换固定扩展
- 法规名额分配零延迟开销，防止通用法规垄断候选
- 两者互补：Adaptive-k 控制总量，名额分配保证法规覆盖
- 已测试 SAR 残差融合失败，这两个方案更简单更直接

**最小实现**：
```
1. Level 1 检索 top-40（与现在扩展到 30 类似）
2. Adaptive-k 截断：找最大分数落差，保留截断前的候选
3. 法规名额分配：每部法规最多 3 个章节
4. 截断后的候选进入 Level 2 条级检索
```

### 优先级

1. **问题B 先做**（Adaptive-k + 名额分配）：零延迟开销，改动最小，可立即验证
2. **问题A 后做**（查询分解 + 多轮检索）：需要新增 LLM 调用和流程编排，改动较大
