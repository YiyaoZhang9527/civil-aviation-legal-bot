## 方案 A：拒答门控 + Set-Membership 降级

### 借鉴自

1. **Lexis+ AI / Westlaw 拒答机制**（2026-02 Stanford Magesh 评测披露的"商用法治 RAG 17-33% 幻觉率"反向推动）：商用旗舰在置信度不足时主动拒绝
2. **Stanford HAI 四类法律 AI 错误分类**（2025）：错误权威（Incorrect Authority）= 引用了不相关法规，本方案直接对治
3. **架构审查 P0-1（空证据 hard gate）+ P0-2（Set-Membership 降级）**：本地已识别的最高 ROI 改造点

### 解决的失败模式

- **CAT4 LLM 过度自信 (8题含 hallucination)**：
  - Q09, Q11, Q41, Q47, Q70, Q80, Q87（5/8 是"编造数字/期限"型，是最危险的 hallucination）
  - Q12（4条 hallucinated，最严重个案）
- **CAT2 检索错配 (5题) 中的"看似有引用但实际错"**：Q03, Q10, Q13, Q27, Q73
  - 当前 Set-Membership 失败时只加"⚠️待核实"警告，用户难以识别
  - 改为 source 降级后自动归到【补充说明】区，与【法律依据】区物理隔离

## 方案 B：CRAG 检索质量补救

### 借鉴自

1. **Corrective-RAG (CRAG)** — Yan et al., 2024（arXiv 2401.15884）：检索后用轻量评估器判断质量，低质量时触发 query 改写/扩展检索
2. **LeMAJ** (arXiv 2510.07243)：基于法律数据点分解的反向验证思路
3. **架构审查 P1-1（合并 SubjectAgent + IssueAgent）的反向应用**：当前 5 个 LLM 调用后才发现"证据不足"，CRAG 把"评估"前移到证据到达时

### 解决的失败模式

- **CAT3 检索缺口 (5题)**：Q02（燃油量）, Q12（CCAR-135 飞行员资质）, Q19（拒载理由）, Q49（通航安保）, Q91（定期检查）
- **CAT2 检索错配 (5题)**：部分可通过"问得更具体"补救：Q10（特技飞行）, Q13（无人机空域）, Q27（飞行程序设计）

**新增方法**（在 RetrievalAgent 类内）：
```python
def _assess_retrieval_quality(self, question: str, evidence: list[Evidence]) -> str:
    """轻量 LLM 评估证据质量: sufficient | partial | insufficient"""
    evidence_summary = "\n".join(
        f"- 《{ev.law_title}》{ev.article}: {ev.text[:150]}"
        for ev in evidence[:5]
    )
    messages = [
        {"role": "system", "content": (
            "你是检索质量评估器。判断以下证据是否能回答用户问题。\n"
            "只输出一个 JSON: {\"quality\": \"sufficient\" | \"partial\" | \"insufficient\"}\n\n"
            "判定标准:\n"
            "- sufficient: 证据中有 2 条以上直接回答问题的具体条款\n"
            "- partial: 证据涉及相关法规但只覆盖总则/前言, 缺少具体条款\n"
            "- insufficient: 证据与问题不相关或覆盖严重不足"
        )},
        {"role": "user", "content": f"问题: {question}\n\n证据:\n{evidence_summary}"},
    ]
    try:
        data = self.llm.json(messages)
        return data.get("quality", "sufficient")
    except Exception:
        return "sufficient"
```

**新增配置开关**（config.py）：

#### 改动 2：_answer 阶段触发补救

**文件**：`legalbot/agents.py:1087-1479`（LegalOrchestrator._answer）

**当前流程**（line 1144-1186）：`RewriteAgent.rewrite_queries` → RetrievalAgent.retrieve → 进 decompose

**改动**（在 retrieve 之后, decompose 之前）：

```python
# CRAG 补救: 质量低时, 用 issue 关键词改写重检索

        and getattr(state, 'retrieval_quality', 'sufficient') == 'insufficient'
        and state.plan.legal_issues):
    logger.info("crag/补救", "检索质量不足, 触发关键词改写重检索", "")
    # 用 IssueAgent 的 legal_issues 重新检索
    rescue_queries = state.plan.legal_issues
    for query in rescue_queries:
        hits = search_index_tree(query=query, law_hints=state.plan.law_hints,
                                  article_hints=state.plan.article_hints,
                                  top_k=config.RETRIEVAL_TOP_K, _logger=logger)
        for hit in hits:
            if (hit.law_id, hit.node_id) not in seen:
                evidence.append(hit)
                seen.add((hit.law_id, hit.node_id))
    # 重新评估
    quality = self.retrieval_agent._assess_retrieval_quality(
        state.normalized_question, evidence
    )
    state.retrieval_quality = quality
```

## 方案 C：SAC（Summary-Augmented Chunking）摘要辅助检索

### 借鉴自

1. **德国民法典分块研究**（arXiv 2605.19806, 2025）：法律结构切分 > 固定大小切分 > 语义切分，**Summary-Augmented Chunking (SAC)** 在 LegalBench-RAG 上效果最好
2. **Remedy Legal** (2026.03)：82% → 99.8% 通过段落级分块（~250 token 段落）
3. **架构审查 P2-1（Rule 7/8 收紧）的间接需求**：当前 LLM 看到长条文被"重心化"稀释，SAC 让长条文也能精确匹配

### 解决的失败模式

- **CAT3 检索缺口（深层）**：长条文 (1500+ 字) 的 embedding 被"重心化"稀释，搜"燃油量"时命中长条文的"目的/依据"段而非具体要求
- **CAT2 检索错配（部分）**：总则长条文（如《民用航空法》整章）被命中时，标题相似但内容不相关

| 摘要偏离原文（LLM 摘要幻觉） | 低 | 中 | 摘要只用于检索初筛，LLM 决策看原文 text，摘要偏差最多影响"召回哪个条文"，不直接影响答案 |

## 关键设计决策记录

1. **为什么把 P0-1 + P0-2 合并为方案 A？**
   - 都是"在合成前/合成中拦截错误内容"，逻辑相邻
   - 改动量都很小（< 30 行），合并后 0.5-1 人天可完成
   - 风险性质相同（拒答率上升），便于一起灰度

2. **为什么方案 B 不用 Self-RAG 而用 CRAG？**
   - Self-RAG 需要微调模型，**不适用**（2026 行业调研明确指出）
   - CRAG 是轻量版，**不需要微调**，可在任何 LLM API 上实现
   - 当前 Reflexion 循环只检查"答案质量"，不检查"检索质量"，CRAG 补这个缺口

3. **为什么方案 C 不直接做"段落级切分"（Remedy Legal 风格）？**
   - Remedy Legal 把 85,000 token 文书切成 250 token 段落，**适用于判例文书**（长文本）
   - 民航法规天然按"条"分块，平均条文 500-1000 字，**不需要再切**
   - 当前问题是"长条文被重心化"而非"块太大"，SAC（摘要辅助）比"继续切小"更对症

4. **为什么 3 方案都用"开关控制"？**
   - 符合项目"前向兼容"原则
   - 灰度验证：可关可开，对比 A/B 效果
   - 失败回退：一键 `HARD_REFUSAL_ON_EMPTY_EVIDENCE=False` / `CRAG_ENABLED=False` / `SAC_ENABLED=False` 回到原行为

5. **为什么方案 B 默认关闭而 A/C 默认开启？**
   - A 几乎无风险（只多 1 个 if），且针对最严重的"过度自信"，默认开
   - C 是基础设施级，索引重建成本高，但运行期无额外延迟，默认开
   - B 增加延迟（+1-2s），且涉及 prompt 工程，默认关闭待消融验证
