# P0 LLM幻觉防控调研：后生成校验方案

> 问题：检索已修到每题正确证据12条，但53%引用声明不被证据支持。
> LLM会：过度延伸、张冠李戴、凭记忆补充。
> 约束：延迟增加 <5%，已有 bge-reranker-v2-m3 cross-encoder。

---

## 方案1：Claim-NLI 逐声明校验（RAGAS Faithfulness 模式）

**原理：**
将LLM答案分解为原子声明(claims)，对每条声明用NLI模型判断是否可从证据推导。
faithfulness = 支持的声明数 / 总声明数。低于阈值则删除或标记该声明。

**实现：**
1. LLM生成答案时要求结构化输出（每条claim带引用chunk_id）
2. 后处理：对每条claim，拼接其引用的证据文本作为premise，claim本身作为hypothesis
3. 用cross-encoder（bge-reranker-v2-m3 或 DeBERTa-v3）打分
4. 分数低于阈值 → 删除该claim或标记为"无法验证"

**延迟：**
- 12条证据 x ~5条claim = ~60次推理
- bge-reranker-v2-m3 单次 ~5ms（GPU），总计 ~300ms
- 占总延迟 <3%（当前生成 ~5-10s）

**适用性：高。** 我们已有bge-reranker-v2-m3，无需额外模型。

## 方案2：Set-Membership 引用校验（regulated-rag 模式）

**原理：**
LLM必须结构化输出每条claim及其引用的chunk_id。后生成确定性校验：每个被引chunk_id必须出现在检索结果集中。未引用的claim触发拒绝。

**实现：**
```
post-generation check:
  for each claim in answer:
    if claim.cited_chunk_ids is empty → CITATION_GROUNDING_FAILED
    for each chunk_id in claim.cited_chunk_ids:
      if chunk_id not in retrieved_set → CITATION_GROUNDING_FAILED
```
确定性代码检查，不依赖模型判断。

**延迟：** ~0ms（纯集合运算）

**适用性：中。** 解决"张冠李戴"（引用了不存在的chunk），但不解决"过度延伸"（引用了正确chunk但claim内容超出chunk范围）。需配合方案1使用。

**来源：** [RZ-Logic/regulated-rag](https://github.com/RZ-Logic/regulated-rag)，FDCPA法规QA，12题eval零grounding failure。

## 方案3：声明分解+三分类校验（SURE-RAG / RT4CHART 模式）

**原理：**
将claim-evidence关系从二分类扩展为三分类：Supported / Refuted / Insufficient。
对每条claim：
- 覆盖度(coverage)：claim的信息是否在证据中出现
- 关系强度(relation)：支持/反驳/无关
- 冲突(conflict)：多条证据之间是否矛盾

只有Supported的claim才保留。

**实现：**
1. 声明分解（LLM或规则）
2. 每条claim与每条证据配对，cross-encoder打分
3. 聚合：claim被任一证据Supported → 保留；全部Insufficient → 删除
4. 可选：全局校验——重新用完整证据集验证，避免碎片化误判

**延迟：**
- 局部验证：~300ms（同方案1）
- 全局验证额外：+200ms（一次完整context输入）
- 总计 ~500ms，占比 <5%

**适用性：中高。** 更精确但复杂度更高。适合v2阶段。

**来源：**
- SURE-RAG (arxiv 2605.03534)：HotpotQA-RAG v3上Macro-F1=0.9075，比GPT-4o judge高17%
- RT4CHART (arxiv 2603.27752)：三分类标签 Entailed/Contradicted/Baseless，RAGTruth++上F1=0.776

## 方案4：轻量NLI分类器（HHEM模式）

**原理：**
用专用幻觉检测模型（如Vectara HHEM-2.1-Open，T5架构）直接对(证据, 答案)对做二分类：一致/幻觉。无需LLM-as-judge。

**实现：**
1. 拼接所有证据为premise，完整答案为hypothesis
2. HHEM-2.1-Open 前向推理，输出一致/不一致概率
3. 低于阈值 → 整体拒绝或触发逐声明审查

**延迟：**
- HHEM-2.1 单次 ~50ms（GPU），~1.5s（CPU）
- RTX 3090上 ~0.6s（2k token输入）

**适用性：低。** 仅给出整体分数，无法定位哪条claim有问题。适合作为快速拒答门控，不适合精细修正。需引入新依赖。

**来源：** [Vectara HHEM-2.1](https://www.vectara.com/blog/hhem-2-1-a-better-hallucination-detection-model)

## 方案5：多阶段确定性架构（regulated-rag 综合模式）

**原理：**
将幻觉防控分层：生成前拒绝 + 结构化输出 + 生成后确定性校验 + 语义充分性校验。
LLM被包围在确定性检查之间，随机性被严格约束。

**架构：**
```
Query
 ├─ 域外预检（regex，确定性）       → 拒绝或继续
 ├─ 检索 + 重排（确定性）
 ├─ 低置信度拒绝（top-1分数 < 阈值）  → 拒绝或继续
 ├─ LLM生成（结构化输出：claim+chunk_ids）
 ├─ Set-membership校验（确定性）      → 引用不存在则拒绝
 ├─ Claim-NLI逐声明校验（cross-encoder）→ 删除不支持的claim
 └─ 输出（仅保留通过校验的claim）
```

**延迟：** 确定性检查 ~0ms，NLI校验 ~300ms，总计 <3%

**适用性：高。** 这是我们应该采用的完整方案。

**来源：** regulated-rag BUILD-DECISIONS.md 的核心原则：
> "Citation grounding and refusal-on-low-confidence belong in code, not in prompts.
> The stochastic stage is bounded by deterministic checks on both sides."

## 方案6：答案修正流程（Delete-Unsupported 模式）

**原理：**
检测到不支持声明后，不是拒绝整个答案，而是自动删除不支持的声明，保留通过校验的部分。避免"全有或全无"。

**实现：**
1. 通过方案5的校验流程，得到每条claim的支持/不支持标记
2. 删除不支持的claim
3. 如果剩余claim不足以回答问题 → 返回"无法确定" + 保留部分信息
4. 如果剩余claim仍构成完整答案 → 直接返回

**关键设计：**
- 删除而非重写：避免二次幻觉
- 保留通过的证据引用：用户可追溯
- 标记哪些信息被删除：可选的透明度功能

**延迟：** ~0ms（在方案5的基础上无额外开销）

**适用性：高。** 配合方案5使用。

## 推荐方案：方案5 + 方案6 的组合

### 核心架构

```
生成前门控（已有）：
  ├─ 域外查询拒绝
  └─ 检索置信度检查

LLM生成（改进）：
  └─ 要求结构化输出：每条claim带chunk_ids引用

生成后校验（新增）：
  ├─ Step 1: Set-membership校验（确定性，~0ms）
  │   └─ 引用的chunk_id必须存在于检索结果集
  ├─ Step 2: Claim-NLI逐声明校验（bge-reranker-v2-m3，~300ms）
  │   └─ 每条claim与其引用的证据做cross-encoder推理
  │   └─ 低于阈值 → 标记为unsupported
  └─ Step 3: 答案修正
      └─ 删除unsupported claims
      └─ 若剩余claim为空 → 返回"根据现有资料无法回答"
      └─ 否则 → 返回修正后的答案
```

### 为什么选这个方案

1. **用已有模型。** bge-reranker-v2-m3 本身就是cross-encoder，可直接做NLI推理，无需引入新依赖。
2. **延迟可控。** 总新增延迟 ~300ms，占当前5-10s生成时间的 <3%。
3. **确定性优先。** set-membership是纯代码检查，不依赖模型；NLI是补充层的语义校验。
4. **非破坏性。** 删除不支持声明而非重写，避免二次幻觉。
5. **可审计。** 每条claim都有支持/不支持标记，可追溯。

### 实现优先级

1. **P0（立即）：** 结构化输出 + set-membership校验。零延迟，解决"张冠李戴"。
2. **P1（本周）：** Claim-NLI逐声明校验 + 答案修正。~300ms延迟，解决"过度延伸"和"凭记忆补充"。
3. **P2（可选）：** 三分类校验（Supported/Refuted/Insufficient），更精细但复杂度更高。
