# 法律引用一致性深度分析报告

> 测试日期: 2026-05-31
> 测试问题: 「不在《运行规范》的机场是否可以去备降」
> 测试轮数: 10
> LLM后端: DeepSeek (deepseek-v4-flash), temperature=0.0
> 检索管线: BM25(35%) + 向量(35%) + 关键词hint(30%) RRF融合, top-k=8
> 自检配置: MAX_REFLEXION_ITERATIONS=2

---

## 一、测试概述

本测试通过向民航法律问答系统重复提交完全相同的问题10次，逐层分析流水线各环节的输出一致性。测试的目标不是评估"答案对不对"，而是评估"同样的输入，系统给出的回答是否稳定"——这是法律问答系统最基本的可靠性要求。

**流水线架构（单轮调用链）：**

```
用户问题
  → SubjectAgent (LLM: 主体提取)
  → IssueAgent (LLM: 争点识别)
  → ClarificationAgent (规则: 澄清判断)
  → RewriteAgent (LLM: 查询改写 → queries / law_hints / article_hints)
  → DecompositionAgent (LLM: 问题拆分)
  → RetrievalAgent (BM25 + 向量 + hint → RRF融合 → top-8 → 回读原文)
  → CrossRef (规则: 交叉引用补全)
  → CitationAgent (LLM: 主张抽取 + 逐条校验)
  → ConflictAgent (规则: 法律层级检测)
  → SynthesisAgent (LLM: 答案生成)
  → ReflexionAgent (LLM: 质量评估 → 若gap则补搜重试, 最多2轮)
```

每轮至少涉及 **8-12次 LLM API 调用**，任何一次调用的非确定性输出都会级联影响下游。

#### 4.1.3 检索层根因分析

检索管线是三路融合结构：`BM25(0.35) + 向量(0.35) + 关键词hint(0.30)`，三路分别独立打分后用RRF融合取top-8。三个环节的非确定性来源：

**A. LLM查询改写的不确定性**

RewriteAgent每次生成的queries、law_hints、article_hints不同。即使是同一个问题，LLM可能生成：
- 轮A: queries=["备降机场 运行规范 要求", "不在运行规范中的机场 备降"]
- 轮B: queries=["备降场选择标准", "运行规范 机场列表 备降"]

不同的queries导致BM25命中的法条完全不同。

**B. top-k截断的边界效应**

取top-8条证据，但许多法条的BM25/向量分数非常接近。第8名和第9名可能只差0.01分，一轮被纳入、一轮被排除。

**C. 第8轮灾难性偏离的具体原因**

轮8的LLM查询改写偏向了"空中交通管理"方向，导致：
- 检索到的12条证据全部来自《民用航空空中交通管理规则》和机场安保规则
- 与"运行规范"和"备降机场选择"完全无关
- SynthesisAgent拿到的证据里没有任何CCAR-121部的内容
- 答案只能诚实回答"无法给出结论"

这是典型的**级联失败**: 一个环节的输出偏离，导致后续所有环节都基于错误的前提工作。

### 4.2 第二层：引用校验（Citation）— 同一法条判定矛盾

#### 4.2.1 校验状态矛盾统计

在61个被校验的node_id中，**7个法条在不同轮次被赋予了不同的校验状态**：

| 法条 | supported | partial | unsupported | 平均置信度 |
|------|-----------|---------|-------------|-----------|
| 121.715 | 3轮 | 1轮 | 5轮 | 0.48 |
| 121.655 | 4轮 | 2轮 | — | 0.83 |
| 121.643 | 4轮 | 1轮 | — | 0.86 |
| 121.641 | 1轮 | 3轮 | 1轮 | 0.70 |
| 121.642 | — | 5轮 | 1轮 | 0.61 |
| 121.663 | — | 1轮 | 4轮 | 0.44 |
| 121.637 | — | 6轮 | — | 0.57 |

最极端的案例是 **CCAR-121.715**：在3轮中被判定为"支持"，1轮"部分支持"，5轮"不支持"。同一条法律，面对同一个问题，LLM的判断从"直接支持"到"完全无关"都有可能。

#### 4.2.2 根因分析

CitationAgent 的校验流程是：
1. LLM从问题中抽取claims（法律主张）
2. LLM逐条判断每条evidence是否支持某个claim
3. 输出 status: supported / partial / unsupported

非确定性来源：
- **claims抽取不稳定** — 不同轮次LLM可能抽取不同的claims。比如轮A抽取"备降机场必须在运行规范中"，轮B抽取"不在运行规范中的机场能否备降"，这两个claim的方向不同，导致同一法条的校验结果不同
- **LLM判断本身不确定** — 即使claims相同，LLM对"是否支持"的判断也有随机性。temperature=0.0不等于确定性——DeepSeek服务端的GPU调度、KV cache命中、batch组合等因素都会影响输出
- **上下文长度影响** — 12条evidence + claims的prompt较长，LLM在长上下文中的注意力分布可能不同

### 4.3 第三层：答案生成（Synthesis）— 格式纪律缺失

#### 4.3.1 文本法条引用提取结果

10轮中只有3轮在答案文本里产生了可被正则 `《法名》...第X条` 解析的格式引用：

| 轮次 | 可解析的法条引用 |
|------|-----------------|
| 1-3 | 无 |
| 4 | 《中华人民共和国民用航空法》第四十七条, 《民航法》第四十七条 |
| 5-6 | 无 |
| 7 | 《运输机场运行安全管理规定》第二十八条 |
| 8 | 无 |
| 9 | 《运输机场运行安全管理规定》第二十八条 |
| 10 | 无 |

**只有3/10轮的答案中包含明确的法条引用**（可被程序化提取）。

#### 4.3.2 根因分析

SynthesisAgent的system prompt要求："回答格式：【结论】、【法律依据（必须引用原文标出出处）】"。但LLM对格式遵守严重不一致：

1. **格式变体多** — 有时写"根据《大型飞机公共航空运输承运人运行合格审定规则》第121.643条"，有时写"第121.643条规定了..."，有时只写法条内容不写条号
2. **正则覆盖不足** — 当前正则 `《([^》]+)》[^。；\n]*?(第[一二三四五六七八九十百千\d]+条)` 只能匹配中文数字条号，无法匹配"第121.643条"这样的阿拉伯数字+小数点格式
3. **LLM倾向意译** — LLM倾向于用口语化方式解释法条内容，而不是直接引用原文条号

值得注意的是，几乎所有轮次的答案都引用了CCAR-121部的法条内容（只是没用标准格式），所以**内容层面的引用实际上比格式层面多得多**。但这仍然是一个可靠性问题——如果下游需要程序化提取引用，当前系统无法满足。

### 4.4 第四层：Reflexion自检 — 负面作用大于正面

#### 4.4.1 自检触发情况

10轮全部触发满额2轮自检（MAX_REFLEXION_ITERATIONS=2），无一轮在第1轮自检中通过。

这意味着每轮的实际API调用次数是：
- 基础流水线: 7-8次调用（主体提取 + 争点识别 + 查询改写 + 引用校验 + 答案生成 + 冲突检测）
- 自检第1轮: 2次调用（质量评估 + 补搜答案生成）
- 自检第2轮: 2次调用（质量评估 + 补搜答案生成）
- **总计: 11-12次API调用/轮**

#### 4.4.2 自检的副作用

1. **每轮额外增加约120秒**（占总耗时的50%以上）
2. **补搜引入新证据**，导致证据组合更不稳定
3. **补搜后重新生成答案**，答案风格和引用方式可能与第一轮不同
4. **从未通过自检**，说明自检标准可能过高或者自检Agent本身不可靠

## 五、非确定性的级联效应

将上述四层的不确定性串联起来，可以得到完整的级联失败路径：

```
LLM生成不同queries (第1层不确定性)
  → 检索到不同法条 (证据组合10/10不同)
    → 引用校验面对不同证据 (第2层不确定性)
      → 同一法条校验状态矛盾
        → LLM基于不同校验结果生成不同答案 (第3层不确定性)
          → 答案长度波动94%, 引用格式不可预测
            → 自检判定"有缺陷", 补搜引入更多不确定性 (第4层不确定性)
              → 最终答案: 60%严格一致, 10%严重偏离
```

**核心结论: 整条流水线的非确定性源头是LLM查询改写。** 如果改写出不同的queries，后续所有环节都会不同。这就像一棵树——根部的微小倾斜，到树梢就变成了巨大的偏移。

## 七、结论与改进方向

### 7.1 现状评估

| 维度 | 评分 | 说明 |
|------|------|------|
| 核心结论一致性 | 6/10 (60%) | 严格一致的只有6轮 |
| 结论方向一致性 | 9/10 (90%) | 排除严重偏离后方向大致正确 |
| 检索层稳定性 | 0/10 (0%) | 10轮10种不同证据组合 |
| 引用校验稳定性 | 低 | 7个法条校验状态矛盾 |
| 答案格式稳定性 | 3/10 (30%) | 仅3轮有可解析的法条引用 |
| 自检有效性 | 0/10 (0%) | 从未通过自检，且引入额外不确定性 |

### 7.2 不一致性的三个根源

**根源1: LLM查询改写是非确定性源头（影响权重: 50%）**

同一问题经LLM改写后产生不同的queries/hints，直接导致检索到不同法条。这是整个级联效应的起点。LLM本身在temperature=0.0下仍具有非确定性（服务端因素），且10轮调用中每次的上下文都不同（因为流水线中上游Agent的输出也在波动）。

**根源2: top-k截断放大微小差异（影响权重: 30%）**

RRF融合后，许多法条分数非常接近。取top-8时，边界法条（排名第7-10位）被随机纳入或排除。12条证据中有4-5条可能是"边缘"证据，不同轮次有不同的边缘证据被选中。

**根源3: Reflexion自检引入额外不确定性（影响权重: 20%）**

自检从未通过，每次都补搜并重新生成。补搜的法条又和原始证据不同，答案生成LLM面对不同的证据组合自然产生不同答案。

### 7.3 可能的改进方向（仅供参考，不构成代码修改建议）

1. **检索去LLM化** — 对同一问题，先做确定性的关键词映射（"备降"→CCAR-121.637/641/642/643/655），再让LLM只负责解读，不负责决定搜什么
2. **缓存检索结果** — 相同问题+相同queries应该命中缓存，避免重复检索
3. **降低Reflexion频率** — 当前MAX_REFLEXION_ITERATIONS=2且从未通过，说明阈值设置有问题。可以考虑只在citation全部unsupported时才触发自检
4. **强制答案格式** — 在SynthesisAgent的prompt中用few-shot示例规定法条引用格式，或后处理提取法条编号
5. **正则升级** — 当前正则无法匹配"第121.643条"格式，需要扩展
